# GPT Image Playground 国际化（i18n）支持需求文档

## 1. 背景与现状

当前项目已经是一个 Web + Tauri Desktop + Mobile/Android 的多运行时工作台，但界面语言仍是单一语言驱动，且文案分散在各个组件中：

- `src/app/layout.tsx` 仍硬编码 `lang='en'`。
- `src/lib/config.ts` 还没有界面语言配置项。
- settings、主工作台、历史、同步、分享、桌面端提示等文案大量直接写在组件里。
- 现有设置面板虽然有“系统配置”主视图，但没有统一的“界面语言”入口。

这会带来几个直接问题：

1. 首次启动无法根据用户语言自动显示合适语言。
2. 用户没有明确的语言切换入口。
3. 后续新增语言时，文案维护成本会快速上升。
4. `html lang`、页面标题、placeholder、ARIA 文案与实际语言不一致，会影响可访问性和浏览器行为。

## 2. 目标

### 2.1 V1 目标

1. 首版只支持 `zh-CN` 和 `en-US` 两种显示语言。
2. 首次启动时，依据运行时可获得的用户语言信息自动选择显示语言。
3. 用户可以在系统设置面板的“常规”区域手动切换界面语言。
4. 切换后当前会话立即生效，刷新或重启后仍保持。
5. 所有用户可见文案、提示、错误、空状态、tooltip、ARIA 标签、页面标题都随语言切换。
6. 语言偏好进入现有配置体系，并在需要时参与云同步。

### 2.2 非目标

首版不做以下内容：

- 不做 `/en`、`/zh` 这类 URL 路由改造。
- 不做繁体中文、日文等第三语言。
- 不做用户自定义翻译编辑器。
- 不翻译用户输入、AI 生成结果、模型/供应商 ID、URL、代码片段。
- 不把 `docs/` 里的手册和规划文档一并国际化。

## 3. 语言策略

### 3.1 语言码

内部建议只保留两个规范值：

- `zh-CN`
- `en-US`

### 3.2 归一化规则

| 输入来源 | 归一化结果 |
| --- | --- |
| `zh`、`zh-CN`、`zh-SG`、`zh-Hans`、`zh-Hant`、`zh-TW`、`zh-HK` | `zh-CN` |
| `en`、`en-US`、`en-GB`、`en-AU` 等 | `en-US` |
| 其他值或空值 | 走默认推断，再不行回退 `zh-CN` |

首版只做简体中文和英文，不单独拆分繁体中文。

### 3.3 首次启动默认值

优先级如下：

1. 已保存的用户语言设置。
2. 运行时检测到的浏览器/系统语言。
3. `zh-CN` 兜底。

### 3.4 运行时语言来源

- Web：优先读取 `navigator.languages`，再读取 `navigator.language`。
- Tauri Desktop：优先读取系统语言信息；如果无法稳定获取，再退回浏览器语言能力。
- Mobile/Android：同样按运行时语言能力检测，不引入平台专属交互。

## 4. 配置与持久化

### 4.1 配置项

在 `AppConfig` 中新增界面语言配置项，建议命名为 `appLanguage`。

### 4.2 默认配置

- `DEFAULT_CONFIG` 必须提供语言默认值。
- 老用户的历史配置缺少该字段时，加载时要自动补齐，不破坏其他配置。

### 4.3 保存与同步

- 语言偏好与现有本地配置同级保存。
- 如果用户开启了 `appConfig` 同步范围，语言偏好应随配置同步。
- 语言切换后应通过现有配置变更机制刷新当前界面，不要求手动刷新页面。

## 5. 设置面板需求

### 5.1 入口位置

- 在系统设置面板主视图新增“常规”区域。
- 界面语言入口必须在首屏可见，不应埋到高级设置或桌面专属设置里。

### 5.2 交互

- 提供语言下拉或分段选择控件。
- 中文界面中显示 `简体中文`、`English`。
- 英文界面中显示 `Chinese (Simplified)`、`English`。
- 用户保存后立即生效。
- 重置配置时回到当前运行时检测得到的默认语言。

### 5.3 反馈

- 当前语言与默认语言可以用轻量状态提示展示，但不能占据过多空间。
- 切换语言不能丢失当前正在编辑的提示词、设置项或弹窗状态。

## 6. 文案覆盖范围

### 6.1 必须本地化的内容

- 顶层导航、按钮、菜单、标签、选项、说明文字。
- 表单 placeholder、helper text、验证提示、错误提示。
- tooltip、popover、确认弹窗、notice/toast。
- 空状态、加载中、状态徽标、统计数字旁的描述。
- settings、history、sharing、sync、desktop-only 提示。
- 页面 title、description、ARIA label、alt text。

### 6.2 需要保留原样的内容

- 用户输入的 prompt、备注、历史记录正文。
- 模型名、供应商名、API URL、文件路径、代码片段。
- 供应商返回的原始错误详情和调试信息。
- AI 生成内容本身，除非用户显式要求翻译。

### 6.3 生成内容边界

- 应用内部的默认系统提示词、预设说明、按钮文案必须本地化。
- 模型输出内容不自动翻译，不做二次改写。

## 7. 架构要求

### 7.1 统一资源层

- 所有用户可见字符串必须从统一语言资源读取，禁止在组件内散落硬编码。
- 语言资源应支持稳定 key、插值参数和数量变化。

### 7.2 语言切换

- 切换语言后，整个应用 shell 需要重新渲染到目标语言。
- 尽量避免 full reload；至少要保证当前输入和本地状态不丢。
- `html lang` 必须与当前显示语言一致。

### 7.3 格式化

- 日期、时间、数字、百分比、文件大小等应使用当前语言对应的格式化规则。
- 不要再用固定中文格式拼字符串。

### 7.4 缺失兜底

- 任何缺失翻译项都必须有明确兜底，不允许空白 UI。
- 开发和测试环境应能暴露缺失 key。
- 发布前应尽量做到 `zh/en` 两个语言包覆盖一致。

## 8. 运行时约束

- 语言策略必须在 Web、桌面端、移动端都可用。
- 不能依赖 Node-only 服务端逻辑。
- 不能因为语言切换破坏静态桌面导出可访问性。
- 首次渲染阶段应尽量避免“先显示错语言，再切换”的闪烁。

## 9. 验收标准

1. 浏览器/系统语言为中文时，首次进入默认显示中文。
2. 浏览器/系统语言为英文时，首次进入默认显示英文。
3. 用户手动切换语言后，刷新页面和重启应用仍保持。
4. 修改语言后，设置面板、主工作台、历史、同步、分享、桌面提示等核心区域都切换到同一语言。
5. `html lang`、页面标题和主要 ARIA 文案与当前语言一致。
6. 长英文文案在移动端和桌面端都不溢出、不遮挡、不压坏布局。
7. 未知语言输入会稳定回退到中文，不会报错或出现空白。

## 10. 测试建议

- 单元测试：语言归一化、默认值推断、配置读写、迁移兜底。
- 组件测试：设置面板语言选择、切换后的文案刷新。
- 端到端检查：Web/桌面、浅色/深色、移动/桌面宽度。
- 人工验收：首次启动中文/英文、手动切换、重置配置、异常回退、长文案溢出。
- 资源校验：`zh/en` 两个语言包 key 覆盖率对齐，避免只翻一半。

## 11. 分期建议

### Phase 1

- 加语言配置与默认检测。
- 接入统一翻译资源。
- 完成设置面板语言开关和 app shell。

### Phase 2

- 覆盖主工作台、历史、同步、分享、桌面提示、管理页等核心文案。
- 补齐日期/数字格式化与错误提示。

### Phase 3

- 全量文案审校、长字符串布局修正、漏翻检查、截图验收。
