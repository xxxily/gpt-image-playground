# 展示组可扩展约束与域名约束显示需求文档

## 1. 背景

当前展示内容体系已经支持展示位、全局展示组、分享展示组、展示素材、展示比例和分享 Profile。公共读取时按照“分享展示组 > 全局展示组 > 旧环境变量展示内容”的优先级，为每个展示位选择当前有效的展示组。

现有展示组已经有绝对时间窗约束：`startsAt` / `endsAt` 决定展示组是否在某个日期范围内可被公共读取接口选中。现在需要新增域名约束，后续还可能继续增加每日时间段、星期几、语言、渠道、运行环境、实验分组等可选约束。如果每增加一种约束都直接在展示组表单、列表、公共前端展示组件和公共接口里写专用判断，后续维护成本会快速上升。

当前最直接的问题是：展示组不区分访问域名。同一个部署如果同时绑定多个域名，例如品牌站、渠道站、测试站或客户专属站，当前所有域名都会看到同一套展示内容。这会带来几个实际限制：

- 管理员无法在同一个服务中为 `site-a.example.com` 和 `site-b.example.com` 配置不同顶部横幅。
- 管理员无法只在测试域名验证新展示组，而不影响正式域名。
- 分享展示组虽然可以通过 `promoProfileId` 区分用户或渠道，但同一个 `promoProfileId` 在所有域名下都会生效，缺少站点边界。
- 短链和多域名部署已经会跟随当前请求域名生成或访问，但推荐展示内容没有对应的域名分流能力。

因此，本需求不应只把域名做成一个孤立字段，而应该先建立展示组级的“可扩展约束体系”：现有绝对时间窗继续兼容，新增域名约束作为第一类可选约束接入同一套评估、摘要、审计和后台展示模型。公共读取接口统一根据请求上下文评估约束，只返回匹配当前上下文的展示内容；公共前台展示组件不需要知道具体有哪些约束类型。

## 2. 目标

1. 全局展示组和分享展示组都支持可扩展的可选约束集合。
2. 域名约束作为本期新增约束类型接入该集合。
3. 默认行为保持不变：未配置额外约束的展示组在所有域名和当前有效时间窗内生效。
4. 约束属于展示组，不属于展示位，也不属于单张素材。
5. 同一个展示位下可以同时存在全域展示组和域名专属展示组；请求某个域名时优先展示更匹配当前域名的展示组。
6. 分享展示组在域名或其他约束不匹配时视为不可用，并按现有规则降级到匹配当前上下文的全局展示组。
7. 短链、普通分享、加密分享不需要新增域名参数；展示内容由最终访问页面发起的 `/api/promo/placements` 请求上下文决定。
8. Tauri 桌面端远程读取展示内容时，域名匹配以被请求的展示服务端点域名为准，并在不支持该能力的旧服务端上安全降级。
9. 后台提供清晰的输入、校验、预览、列表展示、审计和筛选能力，方便管理员管理多域名投放。
10. 后台列表、详情页、素材页和公共前台展示组件应通过通用约束摘要兼容未来新增约束，不要求每新增一种约束就修改公共前台展示逻辑。
11. 约束体系不能削弱现有 URL 安全、分享密钥、短链 allowed origins、secret masking 和后台权限规则。

## 3. 非目标

- 本期不一次性实现所有约束类型；V1 只要求域名约束落地，并为后续约束保留统一模型和展示能力。
- V1 不支持按 URL 路径、查询参数、语言、地区、设备、Referer 或用户身份分流展示内容。
- V1 不支持黑名单式“排除指定域名”；只支持空约束或 allowlist。
- V1 不支持素材级域名约束。同一展示组内所有素材共享同一域名约束。
- V1 不做 DNS 所有权验证，也不证明管理员填写的域名一定已解析到当前服务。
- 域名约束不是访问控制或安全隔离能力。它用于展示内容分流；如果部署允许任意 Host 头直接访问源站，仍需要在网关层限制可信域名。
- 不改变短链目标 URL 校验、短链 allowed origins、分享 Profile ID 公开读取语义。
- 不新增第三方广告投放、AB 实验、频控、曝光统计或转化归因。
- 不承诺复杂未来约束完全零代码上线。要求是：公共前台展示不感知约束类型；后台通用列表/摘要能显示未知或新增约束；复杂编辑器可通过能力元数据或后续自定义编辑组件增强。

## 4. 核心概念

### 4.1 当前请求域名

“当前请求域名”指服务端处理 `/api/promo/placements` 时，从请求 Host 信息解析出的域名。

解析顺序建议复用现有请求来源工具：

1. 可信反向代理转发的 `x-forwarded-host` 和 `x-forwarded-proto`。
2. 请求 `host`。
3. `request.url` 中的 host 作为兜底。

最终匹配使用 host 维度，而不是完整 URL：

- `https://site-a.example.com/path?x=1` 归一化为 `site-a.example.com`。
- `https://site-a.example.com:8443/path` 可归一化为 `site-a.example.com:8443`。
- scheme、path、query、fragment 不参与展示组匹配。

### 4.2 可扩展约束集合

展示组应拥有一个统一的约束集合。公共读取接口在选择候选展示组时，把请求上下文交给约束评估器；只有约束集合整体通过的展示组才可参与后续素材过滤和排序。

推荐抽象：

```typescript
type PromoConstraintSet = {
    version: 1;
    logic: 'all';
    constraints: PromoConstraint[];
};

type PromoConstraint<TType extends string = string, TPayload = unknown> = {
    id: string;
    type: TType;
    enabled: boolean;
    label: string;
    summary: string;
    payload: TPayload;
};
```

V1 规则：

- `logic` 固定为 `all`，即所有启用约束都必须通过。
- 约束只决定展示组是否可参与候选选择，不改变展示素材自身数据。
- 未知约束类型在公共评估时必须按“不通过”处理，避免把本应受限的展示组错误放大到所有用户。
- 未知约束类型在后台编辑时可以只读展示，不能被静默删除。
- 后台通用展示优先使用服务端返回的 `summary`、`label`、`typeLabel`、`chips` 等摘要字段，而不是前端硬编码每种约束类型的文案。

现有 `startsAt` / `endsAt` 继续保留为展示组基础字段，用于兼容已有数据和现有表单。服务端评估时可以把它们视为一个系统内置的 `absoluteDateRange` 约束，统一进入候选过滤链路；未来新增“每日时间段”“星期几”等约束时，不应再把判断散落在不同查询函数中。

### 4.3 约束评估上下文

约束评估器只使用服务端可信上下文和已有公共查询上下文：

```typescript
type PromoConstraintEvaluationContext = {
    now: Date;
    requestHost: string | null;
    device: 'all' | 'desktop' | 'mobile';
    surface: string | null;
    promoProfileId?: string | null;
    locale?: string | null;
    runtime?: 'web' | 'tauri-desktop' | 'tauri-mobile' | 'unknown';
};
```

规则：

- `now` 由服务端生成，不能由客户端 query 参数覆盖。
- `requestHost` 来自服务端解析到的请求 Host，不能由公开 query 参数覆盖。
- `device` 和 `surface` 可以沿用现有公共查询参数，但只能作为展示筛选或体验优化信号，不能当作安全边界。
- 后续如果增加语言、时区、渠道等上下文，应明确来源是请求头、服务端配置、短链记录还是后台预览参数。

### 4.4 域名约束

域名约束是展示组上的一个可选配置：

```typescript
type PromoDomainConstraintMode = 'all' | 'allowlist';

type PromoAllowedDomainRule = {
    type: 'exact' | 'wildcard';
    host: string;
    port?: number | null;
    label: string;
};
```

含义：

- `all`：默认模式，不限制域名，所有访问域名都可以匹配。
- `allowlist`：只有当前请求域名命中 allowlist 中任一规则时，该展示组才可参与选择。
- `exact`：精确匹配单个域名，例如 `site-a.example.com`。
- `wildcard`：匹配某个域名下的子域名，例如 `*.example.com` 匹配 `a.example.com`、`b.a.example.com`，但不匹配 `example.com` 本身。

### 4.5 匹配优先级

同一展示位、同一 scope 下可能有多个有效展示组。加入约束体系后，候选组排序应先考虑约束匹配强度，再沿用现有更新时间优先策略。

建议顺序：

1. 精确域名命中。
2. 通配子域名命中。
3. 其他未来强匹配约束命中，例如后续可能的渠道专属、实验分组专属。
4. 未配置额外约束的全域展示组。
5. 同一匹配强度下，使用现有 `updatedAt desc, createdAt desc` 规则。

这样可以支持“全域兜底 + 某个域名专属覆盖”的常见模式。例如：

| 展示组 | 域名约束             | 当前访问域名         | 是否参与 | 排序含义                 |
| ------ | -------------------- | -------------------- | -------- | ------------------------ |
| A      | 所有域名             | `site-a.example.com` | 是       | 全域兜底                 |
| B      | `site-a.example.com` | `site-a.example.com` | 是       | 精确命中，优先于 A       |
| C      | `*.example.com`      | `site-a.example.com` | 是       | 通配命中，低于 B，高于 A |
| D      | `site-b.example.com` | `site-a.example.com` | 否       | 域名不匹配               |

## 5. 用户场景

### 5.1 多品牌或多客户站点

同一套应用部署绑定多个域名：

- `brand-a.example.com`
- `brand-b.example.com`
- `customer-1.example.com`

管理员希望每个站点的页面顶部横幅展示对应品牌或客户素材。管理员可以为同一个展示位创建多个全局展示组，并分别配置域名约束。

### 5.2 正式站和测试站分离

管理员希望在 `staging.example.com` 上验证新的展示组、素材比例、点击链接和深色模式效果，确认后再复制或修改为正式域名 `app.example.com` 生效。

### 5.3 分享展示内容按站点隔离

管理员给某个分享 Profile 配置展示组，但只希望它在合作方域名 `partner.example.com` 下展示。用户如果把同一个分享链接改到其他域名访问，分享展示组不显示，页面降级到当前域名匹配的全局展示组。

### 5.4 短链渠道差异

同一个长分享内容可以通过不同短链在不同域名下分发。短链跳转后，最终页面发起展示内容请求时，展示内容按最终访问域名匹配。短链自身不需要保存域名约束副本。

### 5.5 未来时间段约束

管理员希望某个展示组只在每天 09:00-18:00 展示，或只在周一到周五展示。这个能力不需要本期完整实现，但底层约束模型应能承载它：

- `startsAt` / `endsAt` 继续表达绝对日期范围，例如活动从 2026-06-01 到 2026-06-30。
- 未来 `timeWindow` 约束表达每天或每周的细分时间段，例如工作日 09:00-18:00。
- 候选评估时，两类时间约束与域名约束共同组成 `all` 逻辑：日期范围通过、时间段通过、域名通过，展示组才可展示。
- 后台列表不需要等到实现专用编辑器才知道如何显示该约束；服务端返回“工作日 09:00-18:00 / Asia/Shanghai”之类的通用摘要即可。

## 6. 后台管理需求

### 6.1 约束条件区域

新增或编辑全局展示组、分享展示组时，表单应有一个统一的“展示约束”区域，而不是把每种约束散落在不同表单位置。

该区域至少展示：

- 基础有效期：沿用现有开始/结束时间字段。
- 可选约束：本期包含域名约束。
- 约束摘要：保存前能看到当前展示组会在哪些条件下生效。

后台前端应优先通过 `/api/promo/capabilities` 或等价接口获取可用约束类型和表单能力：

```typescript
type PromoConstraintCapability = {
    type: string;
    label: string;
    description?: string;
    editor: 'domainAllowlist' | 'timeWindow' | 'select' | 'multiSelect' | 'readonlyJson';
    defaultEnabled: boolean;
    summaryTemplate?: string;
    maxRules?: number;
};
```

原则：

- 后台列表、详情、素材页只依赖约束摘要渲染，不依赖每类约束的 payload 结构。
- 后台表单可以为常见约束提供专用编辑器；没有专用编辑器的新约束至少能以只读摘要展示，避免编辑旧展示组时把未知约束丢失。
- 简单新约束应尽量通过能力元数据和通用字段编辑器配置，不要求每次新增约束都改展示组列表、素材页和公共前台展示逻辑。
- 公共前台展示组件不读取约束集合，只消费 `/api/promo/placements` 返回的最终展示位和素材。

### 6.2 域名约束表单入口

新增全局展示组和新增分享展示组时，表单增加“显示域名”区域。

默认值：

- 选中“所有域名均显示”。
- 不改变现有展示组创建流程。

可选项：

1. **所有域名均显示**：当前展示组不限制域名。
2. **仅指定域名显示**：管理员填写一个或多个域名规则。

“仅指定域名显示”下提供：

- 域名规则输入框，支持一行一个，也可以粘贴逗号、空格、换行分隔的多个域名。
- “添加当前域名”快捷按钮，使用后台当前请求域名填充。
- 归一化后的域名 chip 列表。
- 每条规则显示匹配类型：精确域名或通配子域名。
- 删除单条规则能力。
- 保存前校验并展示字段级错误。

### 6.3 输入与校验规则

管理员可输入以下形式：

| 输入                              | 保存为                          | 说明                        |
| --------------------------------- | ------------------------------- | --------------------------- |
| `site-a.example.com`              | exact `site-a.example.com`      | 精确域名                    |
| `https://site-a.example.com/path` | exact `site-a.example.com`      | 允许粘贴 URL，但只保留 host |
| `site-a.example.com:8443`         | exact `site-a.example.com:8443` | 带端口时要求端口精确匹配    |
| `*.example.com`                   | wildcard `example.com`          | 匹配子域名，不匹配根域名    |
| `localhost:3000`                  | exact `localhost:3000`          | 允许本地测试                |

校验要求：

- 域名统一转小写，去除首尾空格和末尾点号。
- 支持 IDN 域名，保存前统一为 punycode 或明确的规范化 host。
- 不允许保存 scheme 以外的路径、query、fragment、用户名、密码等 URL 信息。
- 不允许空规则、重复规则、非法端口、非法通配符。
- 通配符只能出现在最左侧完整 label，即只允许 `*.example.com`。
- 不允许 `*`、`*.com`、`*.co.uk` 这类过宽规则，至少需要一个可管理的二级域名基础；公共后缀判断可在 V1 简化为“通配符基础域名至少包含两个 label”，后续再接 Public Suffix List。
- 单个展示组最多保存 50 条域名规则。
- 单条规则原始输入长度不超过 253 字符，chip 展示需要截断但 tooltip 可看全量。
- 当模式为 `allowlist` 时，至少需要 1 条有效规则，否则不能保存。

说明：

- 这些域名规则只是匹配条件，不会触发网络请求，不应复用会禁止内网地址的“公共 Base URL”校验。
- 不能因为允许填写 `localhost` 用于测试，就放宽 provider Base URL、图片代理、短链目标 URL 等已有安全校验。

### 6.4 编辑展示组

编辑全局展示组和分享展示组时：

- 显示当前域名约束配置。
- 允许从“所有域名”改为“仅指定域名”。
- 允许从“仅指定域名”改回“所有域名”。
- 如果展示组已启用且已有素材，缩小域名范围前显示非阻塞提示：保存后该展示组只会在指定域名下展示。
- 不使用浏览器原生 `alert`、`prompt`、`confirm`；提示使用现有 Dialog 或 notice 组件。
- 保存后写入审计日志，包含约束模式、变更前后摘要、规则数量，不记录过长原始输入。

### 6.5 展示组列表

全局展示组和分享展示组列表需要展示约束摘要。列表不能只认识域名约束；未来新增时间段、星期、语言等约束时，应通过服务端摘要直接显示。

建议展示：

- `所有域名`。
- 单个域名 chip，例如 `site-a.example.com`。
- 多个域名时展示前 2 个 chip + `+N`。
- 通配规则用 `*.example.com` 展示。
- 未来时间段约束展示为 `工作日 09:00-18:00`、`每天 20:00-23:00` 等摘要 chip。
- 未知但服务端保留的约束类型展示为 `未支持约束：{label}`，并提示编辑时不会自动删除。
- 鼠标或触控展开时可查看完整规则列表。

列表能力：

- 支持按约束类型和关键词搜索或过滤展示组。
- 搜索 `example.com` 时应命中 `example.com`、`site.example.com`、`*.example.com`。
- 分享展示组列表中，域名约束应与 Profile ID 同时可见，避免管理员只复制了 Profile ID 却忘记它有站点限制。

### 6.6 素材管理页

素材管理页不需要为单张素材增加域名或其他约束字段，但需要在页面顶部展示当前展示组约束摘要：

- “当前展示组：所有域名均显示”。
- “当前展示组：仅在 `site-a.example.com`、`*.example.com` 下显示”。
- 未来存在多个约束时展示为“当前展示组约束：仅在 `site-a.example.com`；工作日 09:00-18:00”。

这样管理员进入素材页时能确认当前素材不会在所有域名生效。

### 6.7 文案与国际化

所有新增后台可见文案必须进入 `src/lib/i18n/*` 现有翻译体系，至少保持中文和英文同步。

域名、Profile ID、展示位 key、URL 示例等技术标识使用 `data-i18n-skip` 或等价方式避免被翻译。

## 7. 公共读取与展示逻辑

### 7.1 API 输入

`/api/promo/placements` 不应接受公开的 `domain`、`host`、`previewHost` 参数来决定正式展示内容。正式匹配必须来自服务端看到的请求 Host，避免普通用户通过 query 参数伪造展示域名。

现有 query 参数继续有效：

- `slots`
- `surface`
- `device`
- `promoProfileId`

路由层需要把服务端解析出的约束评估上下文传入展示内容查询：

```typescript
type PromoPlacementsQuery = {
    slots?: string[];
    surface?: string | null;
    device?: string | null;
    promoProfileId?: string | null;
    context: PromoConstraintEvaluationContext;
};
```

其中 `context.requestHost` 用于域名约束，`context.now` 用于绝对时间窗和未来时间段约束。公开 API 不应允许客户端通过 query 参数覆盖这些服务端可信字段。

### 7.2 候选过滤

公共读取接口按展示位处理时，候选过滤顺序应为：

1. 展示位必须启用。
2. scope 匹配：分享候选只看 `share`，全局候选只看 `global`。
3. 展示组必须启用。
4. 展示组基础绝对时间窗必须通过。
5. 展示组约束集合必须整体通过，包括本期域名约束和未来时间段等约束。
6. 展示素材必须启用、在有效时间窗、匹配设备并且 URL 合法。
7. 至少存在一条有效素材。
8. 候选排序按“约束匹配强度 > 更新时间”。

分享展示组和全局展示组的优先级保持不变：

1. 如果请求带 `promoProfileId`，并且对应分享 Profile 有可用、约束匹配、素材有效的分享展示组，则返回分享展示组。
2. 如果分享展示组不存在、停用、过期、约束不匹配或无有效素材，则继续尝试当前上下文匹配的全局展示组。
3. 如果没有匹配的全局展示组，则 `generation_form_header` 仍可降级到旧环境变量展示内容。

### 7.3 无法解析请求域名

如果服务端无法解析某个约束依赖的上下文字段：

- `all` 展示组仍可匹配。
- `allowlist` 展示组不匹配。
- 依赖缺失上下文的未来约束默认不匹配，除非该约束类型明确声明缺失上下文时允许通过。
- 可以在开发日志中记录一次低噪声 warning，避免线上日志泛滥。

### 7.4 公共响应

V1 公共响应不需要暴露完整域名约束规则，避免向所有访客泄露后台投放配置。

可选返回调试友好的非敏感字段，但公共前端不得依赖它决定展示：

```typescript
type PromoPlacement = {
    slotKey: string;
    source: 'share' | 'global' | 'legacy';
    constraintMatch?: {
        strength: 'exact' | 'specific' | 'all';
        summary?: string;
    };
    // existing fields...
};
```

如果担心响应结构变更影响客户端，`constraintMatch` 可以先不返回，只在服务端内部用于排序。

## 8. 数据模型与迁移

### 8.1 推荐字段

不要只为域名新增单一业务字段。建议在 `promo_configs` 增加一个可扩展约束集合字段：

```sql
ALTER TABLE promo_configs ADD COLUMN constraintsJson TEXT;
```

推荐结构：

```typescript
type PromoConfigConstraintsJson = PromoConstraintSet;

type PromoDomainConstraintPayload = {
    mode: 'all' | 'allowlist';
    rules: PromoAllowedDomainRule[];
};

type PromoTimeWindowConstraintPayload = {
    timezone: string;
    windows: Array<{
        daysOfWeek: number[]; // 1-7, Monday-Sunday
        startTime: string; // HH:mm
        endTime: string; // HH:mm
    }>;
};
```

`constraintsJson` 示例：

```json
{
    "version": 1,
    "logic": "all",
    "constraints": [
        {
            "id": "domain",
            "type": "domain",
            "enabled": true,
            "label": "显示域名",
            "summary": "仅在 site-a.example.com、*.example.com 下显示",
            "payload": {
                "mode": "allowlist",
                "rules": [
                    { "type": "exact", "host": "site-a.example.com", "port": null, "label": "site-a.example.com" },
                    { "type": "wildcard", "host": "example.com", "port": null, "label": "*.example.com" }
                ]
            }
        }
    ]
}
```

说明：

- V1 中展示配置数量通常较小，公共接口已经按展示位加载候选配置，JSON 字段足够满足筛选。
- 域名约束不建议再单独保存为 `domainConstraintMode` / `allowedDomainsJson` 的唯一来源，避免后续每加一种约束都继续扩表。
- 如需优化域名搜索、列表筛选或大规模投放，可以增加派生索引字段或规范化表，例如 `promo_config_constraint_index(configId, type, searchKey)`；这些索引只能作为查询优化，不能成为约束事实来源。
- 现有 `startsAt` / `endsAt` 保留在原字段中，作为兼容和常用基础有效期；服务端评估时将它们纳入统一约束评估链路。

### 8.2 兼容策略

历史数据：

- `constraintsJson` 为空时按空约束集合处理，即没有新增可选约束。
- 历史 `startsAt` / `endsAt` 继续生效。
- 不对历史展示组自动增加 allowlist，避免改变现有线上展示。

异常数据：

- `constraintsJson` 损坏时，服务端应按“不通过新增约束”处理并记录 warning，避免把本应受限的展示组放大到所有用户。
- 约束集合里出现未知 `type` 时，公共评估按不匹配处理；后台列表用只读摘要展示。
- `domain.mode = allowlist` 但没有有效规则时，该展示组不参与公共展示；后台列表应显示“域名规则无效”。
- 创建和更新接口不允许保存上述异常状态。

### 8.3 审计

创建和更新展示组时，审计日志需要记录：

- 约束集合摘要，例如 `constraints: domain(2), timeWindow(1)`。
- 变更前后摘要，例如 `domain: all -> allowlist(2)`。
- 每类约束的规则数量和启停状态。
- 前几个规范化规则标签，超出时只记录数量。

审计日志不需要记录用户粘贴的原始 URL path、query、fragment。

## 9. 与现有能力的关系

### 9.1 现有开始/结束时间

现有开始/结束时间是展示组基础有效期，不应在这次改造中删除或改变语义。

处理方式：

- 后台表单可以继续保留开始/结束时间字段，降低迁移成本。
- 服务端候选评估把开始/结束时间作为系统内置约束处理。
- 后续“每日时间段”“星期几”这类细分时间约束必须与开始/结束时间共同生效，而不是替代开始/结束时间。
- 例如：`startsAt=2026-06-01`、`endsAt=2026-06-30`、`timeWindow=工作日 09:00-18:00` 表示只在 2026 年 6 月内的工作日白天展示。

### 9.2 未来约束类型

未来可扩展约束可以包括但不限于：

| 约束类型     | 典型用途                  | 上下文来源                 | V1 要求            |
| ------------ | ------------------------- | -------------------------- | ------------------ |
| `timeWindow` | 每日/每周时间段展示       | 服务端时间 + 配置时区      | 预留模型和摘要展示 |
| `locale`     | 不同语言站点展示不同素材  | 请求头或应用语言           | 预留模型           |
| `channel`    | 不同渠道短链或活动来源    | 短链记录或服务端注入上下文 | 预留模型           |
| `runtime`    | Web、桌面、移动端差异展示 | 客户端/服务端运行时上下文  | 预留模型           |
| `experiment` | 小规模测试或灰度          | 服务端实验分组             | 预留模型           |

新增约束类型时必须满足：

- 公共前台展示组件不改造也能继续显示服务端筛选后的展示位。
- 后台列表和素材页可以通过 `summary` / `chips` 显示约束摘要。
- 服务端必须为新约束提供解析、校验、摘要、评估和审计摘要。
- 如果后台还没有对应编辑器，编辑页必须以只读方式保留该约束，不能保存时丢失。

### 9.3 展示比例

域名约束和展示比例都是展示组级配置，互不覆盖。

匹配流程先选择展示组，再使用该展示组自己的展示比例渲染。不同域名命中不同展示组时，可以自然展示不同素材、不同轮播间隔、不同切换方式和不同展示比例。

### 9.4 分享 Profile

分享 Profile 仍然只是公开读取 ID。域名约束不会改变 Profile ID 的生成、复制和分享方式。

当分享链接携带 `promoProfileId`：

- 当前域名命中该分享展示组域名约束时，显示分享展示组。
- 当前域名不命中时，该分享展示组视为不可用，继续查找当前域名匹配的全局展示组。
- 如果全局展示组也没有命中，则隐藏展示位或使用旧环境变量兜底。

### 9.5 短链

短链不需要新增域名约束字段。

短链打开后，最终访问页面会按自己的域名请求 `/api/promo/placements`，展示内容由该请求域名决定。短链后台已有的推荐内容策略继续保留：

- 继承长链接中的 `promoProfileId`。
- 移除 `promoProfileId`。
- 覆盖为指定分享 Profile。

如果短链绑定了指定 Profile，但最终访问域名不匹配该 Profile 对应分享展示组，展示内容按现有降级链路处理。

### 9.6 Tauri 桌面端

桌面端展示内容读取模式包括当前站点、自定义域名、完整接口和关闭。

域名匹配规则：

- Web 浏览器访问时，匹配用户当前访问 Web 站点的域名。
- Tauri 桌面端读取远程展示服务时，匹配远程 `/api/promo/placements` 请求到达服务端时的 Host。
- 如果桌面端配置的是完整接口，例如 `https://content.example.com/api/promo/placements`，则服务端看到的域名通常是 `content.example.com`。
- 如果希望桌面端按某个品牌域名展示不同内容，应配置对应品牌域名下的展示服务端点，而不是依赖本地静态包 origin。

旧版远程服务端没有域名约束字段时，桌面端继续按旧逻辑展示，不应崩溃。

### 9.7 旧环境变量展示内容

旧环境变量展示内容只作为 `generation_form_header` 的兼容兜底，不增加域名约束能力。

如果部署者需要按域名控制旧横幅，应迁移到后台展示组系统。

## 10. 安全与部署注意事项

### 10.1 Host 头可信度

域名约束依赖服务端看到的 Host。部署者需要保证：

- 反向代理只接收和转发可信域名。
- 源站不要被任意 Host 头直接访问。
- `x-forwarded-host` 和 `x-forwarded-proto` 来自可信代理，而不是不受控客户端。
- 多层代理场景下，应用读取到的 Host 与用户访问域名一致。

如果无法保证这些条件，域名约束仍可用于一般投放分流，但不能被当作安全隔离。

### 10.2 不新增 SSRF 风险

域名规则只用于字符串匹配，不触发 DNS 查询、HTTP 请求或图片探测。

因此：

- 不应该因为这个功能去请求管理员填写的域名。
- 不应该把域名规则传给图片代理、provider Base URL 或短链跳转校验。
- 现有 `validatePublicHttpBaseUrl`、`normalizeOpenAICompatibleBaseUrl`、短链 target URL 校验和图片 URL 校验保持原语义。

### 10.3 信息暴露

公共 `/api/promo/placements` 响应不应返回完整 allowlist 或完整约束集合。后台 API 可以返回完整约束规则，但必须继续走管理员权限校验。

## 11. 验收标准

### 11.1 后台配置

- 新建和编辑展示组时，有统一的“展示约束”区域，能同时展示基础有效期和新增可选约束。
- 新建全局展示组时，可以选择“所有域名均显示”或“仅指定域名显示”。
- 新建分享展示组时，可以选择“所有域名均显示”或“仅指定域名显示”。
- “仅指定域名显示”下至少填写一个有效域名，否则不能保存。
- 支持精确域名、带端口域名、本地测试域名和 `*.example.com` 通配规则。
- 粘贴完整 URL 时只保存 host，不保存 path、query、fragment。
- 展示组列表能清楚看到每个展示组的约束摘要，包括域名约束和未来约束。
- 素材管理页能看到当前展示组的约束摘要。
- 未知或暂不支持编辑的约束类型不会在编辑保存时被静默删除。
- 约束变更写入审计日志。

### 11.2 公共展示

- 未配置域名约束的历史展示组继续在所有域名下展示。
- 同一展示位同时存在全域展示组和当前域名专属展示组时，当前域名专属展示组优先。
- 当前域名没有专属展示组时，使用全域展示组兜底。
- 当前域名与分享展示组不匹配时，即使 URL 带正确 `promoProfileId`，也不显示该分享展示组。
- 分享展示组域名不匹配后，可以降级到当前域名匹配的全局展示组。
- 不同域名访问同一部署，可以在同一展示位看到不同展示内容。
- 请求域名无法解析时，allowlist 展示组不显示，全域展示组仍可显示。
- 公共前台展示组件不读取 `constraintsJson`，只展示 `/api/promo/placements` 返回的最终 placement。
- 后续新增服务端约束类型时，只要公共接口仍返回同样的 placement 结构，公共前台展示不需要为新约束修改匹配逻辑。

### 11.3 兼容

- 历史数据迁移后不需要管理员立即处理，也不会改变现有展示结果。
- 旧环境变量 `generation_form_header` 兜底仍可显示。
- 旧版桌面端或旧版远程展示服务响应不会让页面崩溃。
- 短链创建、跳转和推荐内容绑定语义不变。
- 展示比例、轮播间隔、切换方式、素材设备过滤不因域名约束失效。
- `constraintsJson` 为空、损坏、包含未知约束类型时都有明确降级策略，不会把受限展示组错误扩展到所有域名。

## 12. 测试计划

### 12.1 单元测试

- 约束集合解析：空值、合法 JSON、损坏 JSON、未知类型、禁用约束、`logic=all`。
- 域名归一化：大小写、末尾点号、完整 URL、端口、localhost、IDN/punycode。
- 规则解析：精确域名、通配子域名、重复规则、非法通配符、非法端口、空输入。
- 匹配逻辑：exact、wildcard、all、allowlist 空规则、无法解析请求域名。
- 排序逻辑：exact 优先于 wildcard，wildcard 优先于 all，同级按更新时间排序。
- 时间约束预留：`startsAt` / `endsAt` 作为系统内置约束参与统一评估。
- 未来时间段约束样例：工作日 09:00-18:00 在时区内通过/不通过。
- 异常数据归一化：历史空字段、未知 mode、损坏 JSON。

### 12.2 集成测试

- `/api/admin/promo/configs` 创建全局展示组时保存域名约束。
- `/api/admin/promo/configs` 创建分享展示组时保存域名约束并生成 Profile ID。
- `/api/admin/promo/configs/{id}` 更新域名约束、改回所有域名、非法输入报错。
- `/api/admin/promo/configs/{id}` 编辑包含未知约束类型的历史记录时，未知约束不丢失。
- `/api/promo/placements` 在不同 Host 下返回不同展示组。
- `/api/promo/placements` 在分享 Profile 域名不匹配时降级到全局展示组。
- `/api/promo/placements` 在 exact、wildcard、all 同时存在时选择正确候选。
- `/api/promo/placements` 对 `startsAt` / `endsAt`、域名约束和未来时间段约束按 `all` 逻辑共同评估。
- 短链跳转后最终页面携带 `promoProfileId` 时，展示内容仍按最终访问域名过滤。
- 审计日志包含域名约束变更摘要。

### 12.3 UI 与视觉验证

- 后台新建/编辑全局展示组：light/dark、桌面/移动布局。
- 后台新建/编辑分享展示组：Profile ID 区域与域名约束区域不挤压、不溢出。
- 展示组列表：多域名 chip、通配域名、长域名截断和展开。
- 展示组列表：模拟未来 `timeWindow` 约束摘要时无需新增列表专用 UI。
- 素材管理页：展示组比例和约束摘要同时存在时布局清晰。
- 前台三个展示位在不同域名命中不同展示组时，不影响展示比例和轮播控制。

## 13. 实施顺序建议

1. 新增约束集合类型、约束注册表、通用解析/校验/摘要/评估入口，并把现有 `startsAt` / `endsAt` 接入统一评估链路。
2. 新增域名约束实现：规则解析、归一化、匹配强度、摘要生成和单元测试。
3. 扩展 `promo_configs` schema、SQLite 创建语句和迁移语句，新增 `constraintsJson`，保证历史数据按空约束集合兼容。
4. 扩展后台创建/更新 schema、服务端 create/update/list 逻辑和审计日志，保存时不能丢失未知约束。
5. 扩展 `/api/promo/capabilities`，返回可用约束类型、摘要字段和后台编辑能力。
6. 扩展 `/api/promo/placements` 路由，把服务端约束评估上下文传入 `getPromoPlacements`，并在分享/全局候选选择时应用统一约束过滤和匹配强度排序。
7. 改造展示组表单、展示组列表、素材管理页摘要，并补齐中文和英文 i18n 文案；公共前台展示组件保持只消费最终 placement。
