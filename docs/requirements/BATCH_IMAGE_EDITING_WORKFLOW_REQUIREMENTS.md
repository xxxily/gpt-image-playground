# GPT Image Playground 批量改图与图片集处理能力需求文档

## 实现状态

- 2026-06-03：P0 已落地，包括空 prompt 批量入口、独立待处理图片区、多图片/文件夹导入、统一指令逐图改图、共享参考图、批量改图预览、确认入队、任务/历史输入图元数据和草稿缺失文件保护。
- P1/P2 仍为后续阶段，包括每图多变体、JSON/CSV 按图片映射、批量结果分组、前后对比、批量导出、蒙版批量编辑和复杂配对等能力。

## 1. 背景与问题

现有批量任务主要围绕“批量生图”展开：用户输入一段 prompt、长文本或 JSON 后，系统生成多条可执行任务，再继承当前表单参数和当前源图片进入队列。这能覆盖批量创作、长文本拆分、多版本探索等场景，但对“批量改图”支持不足。

典型问题包括：

- 当前源图片区承担的是单任务图片编辑、参考图、图生文和图生视频等上下文，不适合作为大批量待处理图片列表。
- 图片编辑模型通常有单次源图张数限制，把一批待处理图片全部塞进现有源图片区会超过限制，也会混淆“待改图片”和“参考图”。
- 批量改图需要稳定的一一对应关系：第 1 张输入图应产出第 1 组结果，第 2 张输入图应产出第 2 组结果，历史、导出、重试和文件命名都要能追踪回原图。
- 风格迁移、老照片修复、统一参考图再创作等场景通常只需要一条共享指令或一个操作预设，不应因为主 prompt 为空就无法打开批量入口。
- 当前批量按钮必须有提示词才可用，导致用户无法先打开批量面板再上传图片集、选择批量改图模式和补充任务说明。

因此，批量能力需要从“把文本规划成多条生图任务”扩展为“把一批图片规划成多条图片编辑任务”，并在产品概念、数据结构、上传入口、预览确认、执行队列和历史结果里明确区分两类输入。

## 2. 目标

1. 新增独立的批量待处理图片集入口，不复用现有源图片区承载大批量图片。
2. 支持用户批量上传图片文件，并支持在桌面环境直接拖入文件夹进行批量处理。
3. 支持一张待处理图片对应一条或多条图片编辑任务，执行结果能稳定追踪到原始图片。
4. 支持批量风格迁移、批量老照片修复、批量增强、批量换背景、批量商品图重绘、统一参考图基准下的批量再创作等高频改图场景。
5. 支持共享参考图与批量待处理图同时存在：参考图用于统一风格、角色、品牌、构图或质感，待处理图用于逐张生成改图任务。
6. 支持空 prompt 打开批量面板；有 prompt 时批量按钮保持现有高亮激活状态，无 prompt 时显示为可用的普通按钮。
7. 批量预览必须在确认前展示每张图片的缩略图、文件名、任务 prompt、参考图策略、输出命名和有效状态。
8. Web 与 Tauri 桌面端都要可用；文件夹处理、文件引用、执行请求和失败降级需要分别定义。
9. 不破坏现有批量生图、文本切分、JSON 导入、任务队列、历史、同步和分享禁用批量的既有语义。

## 3. 非目标

- P0 不要求批量视频、批量图生文或批量多模态分析。
- P0 不要求自动理解每张图片内容并为每张图生成完全不同 prompt；可先用共享 prompt、操作预设或导入元数据驱动。
- P0 不要求在 Web 端持久保存本地文件夹权限；刷新后可以要求用户重新选择批量图片。
- P0 不要求支持蒙版批量编辑、局部区域配对、PSD/RAW 等专业格式。
- P0 不要求把所有批量输出自动打包下载；可以先保证历史、任务分组和单项保存可追踪。
- 不允许通过 JSON 或文件夹元数据自动读取任意本地路径、远程 URL 或敏感配置。

## 4. 核心概念

### 4.1 源图片

源图片是当前编辑区已有概念，用于单任务图片编辑、图生文、图生视频或少量参考图输入。源图片数量受模型和现有校验限制，仍保留在主表单里。

在批量改图中，源图片可作为“共享参考图”使用，例如统一画风、统一人物设定、统一品牌视觉、统一构图基准。它不应被当作待批量处理的图片列表。

### 4.2 批量待处理图片

批量待处理图片是新增概念，表示“需要逐张处理的输入图片集”。每张待处理图片默认生成一个批量子任务，也可以在高级模式下生成多个变体。

批量待处理图片需要有独立上传区、独立列表、独立数量限制、独立草稿摘要和独立执行映射。

### 4.3 共享参考图

共享参考图用于影响整批任务的风格、人物、产品、构图、色彩或材质。首版可以直接复用当前源图片区，也可以在批量面板中提供“使用当前源图片作为共享参考”的显式说明。

共享参考图与待处理图片合并进入单条编辑请求时必须有稳定顺序和提示词约束。建议约定：每条任务的第 1 张图片是待处理目标图，后续图片是共享参考图；如果供应商支持角色化图片输入，再按 provider adapter 映射到更明确的 `target` / `reference` 角色。

### 4.4 批量改图计划

批量改图计划是批量预览的结构化数据。它与现有 `BatchPlan` 兼容，但必须增加图片输入映射，不能只用 `sourceImagePolicy: inherit-all` 表达所有图片关系。

## 5. 目标场景

| 场景              | 用户输入                              | 系统行为                           | 关键要求                                |
| ----------------- | ------------------------------------- | ---------------------------------- | --------------------------------------- |
| 批量风格迁移      | 多张照片 + 一条风格 prompt            | 每张照片生成一条图片编辑任务       | 输出与原图一一对应                      |
| 批量老照片修复    | 多张老照片，可无自由 prompt           | 使用修复预设生成每张图片的修复任务 | 无 prompt 也可启动流程                  |
| 批量商品图优化    | 一批商品图 + 背景/灯光/电商规格说明   | 逐张重绘或增强商品图               | 保留主体和商品识别信息                  |
| 统一参考图再创作  | 当前源图片作为参考 + 一批待处理图     | 每张待处理图在统一参考风格下再创作 | 区分参考图与待处理图                    |
| 批量头像/角色统一 | 统一角色或风格参考 + 多张人物图       | 保留每张人物图特征并套用统一视觉   | 明确隐私和人物保留提示                  |
| 批量变体探索      | N 张待处理图 + 每图 K 个版本          | 每张图生成 K 条子任务              | `batchIndex` 与输入图、变体序号都可追踪 |
| 外部任务导入      | JSON 包含图片文件名和 per-item prompt | 与已上传图片按文件名或相对路径匹配 | 缺失、重复、未匹配项可见                |

## 6. 入口与按钮状态

### 6.1 批量按钮可用性

批量按钮不应再依赖主 prompt 非空才能打开。按钮只在以下硬性条件下禁用：

- 当前是图生文、文生视频、图生视频等暂不支持批量图片编辑的模式。
- 分享链接或临时 API Key 策略明确禁用了批量能力。
- 当前配置处于必须先处理的阻塞状态，例如批量能力被管理员禁用。
- 正在执行会导致面板状态不一致的互斥操作，例如当前正在进行批量规划提交且不可重复打开。

主 prompt 为空时：

- 按钮显示为普通可用状态，视觉上接近图生文按钮的中性入口。
- 点击后打开批量面板，允许用户先添加待处理图片、选择批量改图模式或使用预设。
- tooltip 说明“打开批量任务，可以先添加图片或导入任务”。

主 prompt 非空时：

- 按钮显示为现有的可用高亮状态，表示当前 prompt 会被带入批量流程。
- 点击后打开批量面板，并把当前 prompt 作为默认共享指令。

### 6.2 面板默认落点

打开批量面板后应根据上下文选择默认创建方式：

- 有待处理图片草稿时，优先恢复批量改图面板。
- 有主 prompt 但没有待处理图片时，保留现有 AI 规划 / 文本切分 / JSON 导入入口。
- 无主 prompt 且无待处理图片时，默认展示批量待处理图片上传区和批量改图模式选择。
- 当前源图片区有图片时，显示“可作为共享参考图使用”的状态，但不自动把它们解释成待处理图片。

## 7. 批量图片上传与文件夹处理

### 7.1 独立上传区

批量面板需要新增独立的“待处理图片”区域，功能包括：

- 点击选择多个图片文件。
- 拖拽多个图片文件。
- 拖拽文件夹并递归读取其中的图片文件。
- 展示缩略图、文件名、相对路径、尺寸、文件大小、状态和排序序号。
- 支持移除、重新排序、全选、反选、清空无效项。
- 支持按文件名、相对路径、文件大小、添加顺序排序。

该区域不能与主表单源图片区共用状态，否则会导致单任务图片编辑和批量改图互相污染。

### 7.2 文件夹拖入

Web 端：

- 在支持 File System Access API 或 `DataTransferItem.webkitGetAsEntry` 的浏览器中，支持拖入文件夹并递归枚举图片。
- 不支持文件夹读取的浏览器应降级为多文件选择，并显示内联提示。
- Web 端不得保存或上传用户完整本地绝对路径，只可在 UI 中展示相对路径或文件名。

Tauri 桌面端：

- 应提供文件夹选择或拖入文件夹能力，使用 Tauri 文件系统能力枚举图片。
- 文件读取、缩略图生成和执行请求必须经过桌面运行时封装，不在共享浏览器模块中直接导入 Tauri API。
- 如果桌面文件引用可以跨会话恢复，应只保存必要的本地引用摘要；云同步不得同步本地绝对路径。

移动端：

- 可以先支持多图片选择。
- 文件夹拖入不作为移动端 P0 要求，但面板布局必须能处理较长图片列表。

### 7.3 校验与限制

- 复用现有图片格式、尺寸和单图大小校验，并为批量输入增加单独的总数和总大小上限。
- 默认上限建议可配置，例如 P0 单批最多 50 张待处理图片，大批量确认阈值 20 张；具体值在实现时按性能测试调整。
- 无效文件不阻断整批导入，应作为列表项 warning 展示，并允许用户清除。
- 重复文件不自动删除，但应提示可能重复；可按文件名、大小和最近修改时间做轻量判断，避免读取全量字节做哈希。
- 缩略图和尺寸读取必须懒加载或分批处理，不能阻塞首屏或拖拽响应。

## 8. 批量改图模式

### 8.1 统一指令逐图处理

每张待处理图片使用同一条共享 prompt 或同一操作预设，生成一条图片编辑任务。适用于风格迁移、修复、增强、统一背景和商品图优化。

要求：

- 允许用户不输入主 prompt，但必须选择一个内置操作预设，或在确认前补充共享指令。
- 每条任务的 prompt 由共享指令、操作预设和可选 per-item 信息合成。
- 预览中可逐条覆盖 prompt，覆盖后标记 `lockedByUser`。

### 8.2 统一参考图下再创作

用户在源图片区或批量面板参考图区提供一组共享参考图，再上传待处理图片集。系统为每张待处理图片创建编辑任务，并在 prompt 中明确参考图用途。

要求：

- 明确显示当前共享参考图数量和来源。
- 每条任务必须同时携带待处理图片和共享参考图，且顺序稳定。
- 如果当前模型单次最大图片数不足以容纳“1 张待处理图 + N 张参考图”，必须在预览前提示用户减少参考图或切换模型。
- prompt 应明确“第一张为待处理目标图，后续为参考图”的约定，除非 provider adapter 支持更明确的图片角色。

### 8.3 每图多变体

用户可以设置每张待处理图片生成多个版本，例如每张图输出 3 种风格。该模式会产生 `待处理图片数 × 每图变体数` 条任务。

要求：

- 预览和确认按钮必须显示真实任务总数。
- 大批量必须触发二次确认。
- 历史和结果需要能区分同一输入图的第几个变体。
- P0 可以只支持固定每图变体数，P1 再支持每图不同变体数。

### 8.4 per-item 指令导入

用户可通过 JSON 导入或后续 CSV/表格能力为不同图片提供不同 prompt。

要求：

- 导入数据通过 `filename`、`relativePath`、`externalId` 或显式 `inputImageId` 与已上传图片匹配。
- 未匹配、重复匹配、多余图片都必须进入 warnings。
- JSON 不得引用未由用户选择的本地路径自动读取文件。

## 9. 数据结构建议

现有 `BatchPlan` 可继续服务批量生图，但批量改图需要扩展图片输入映射。建议新增或扩展如下结构：

```ts
type BatchImageInputRole = 'edit-target' | 'shared-reference' | 'mask' | 'context-reference';

type BatchImageInputRef = {
    id: string;
    role: BatchImageInputRole;
    filename: string;
    relativePath?: string;
    mimeType: string;
    sizeBytes?: number;
    width?: number;
    height?: number;
    order: number;
    status: 'ready' | 'warning' | 'invalid' | 'missing';
    warning?: string;
};

type BatchImageEditPlan = BatchPlan & {
    taskFamily: 'image-edit-batch';
    batchInputImageCount: number;
    sharedReferenceImageCount: number;
    imagePairingMode: 'one-image-per-task' | 'one-image-multi-variant' | 'json-mapped';
    outputNaming?: BatchOutputNamingTemplate;
    imageInputs: BatchImageInputRef[];
    tasks: BatchImageEditPlanItem[];
};

type BatchImageEditPlanItem = BatchPlanItem & {
    inputImageRefs: string[];
    sharedReferenceRefs: string[];
    maskImageRef?: string;
    inputImageOrder: number;
    inputImageFilename: string;
    inputImageRelativePath?: string;
    variantIndex?: number;
    variantTotal?: number;
    operationPreset?: 'style-transfer' | 'photo-restore' | 'enhance' | 'background-rework' | 'custom';
    outputFilenameHint?: string;
    pairingKey?: string;
};
```

兼容要求：

- 现有 `BatchPlanItem.sourceImagePolicy` 继续保留，用于旧批量规划。
- 新批量改图不得只依赖 `sourceImagePolicy: inherit-all`，必须保存明确的 `inputImageRefs`。
- 草稿、历史和任务状态中的批量元数据必须能记录输入图片 ID、原始文件名和变体序号。

## 10. 预览与编辑

批量改图预览应优先使用列表或紧凑表格，而不是只展示 prompt 卡片。每条预览项至少包含：

- 待处理图片缩略图。
- 文件名和相对路径。
- 图片状态：可用、无效、缺失、尺寸风险、格式风险。
- 当前任务 prompt，可编辑。
- 共享参考图状态。
- 操作预设或变体方向。
- 输出文件名预览。
- 启用 / 禁用状态。
- 删除、复制、上移、下移、重置为批量默认值。

批量级操作包括：

- 应用共享 prompt 到全部未锁定条目。
- 切换操作预设。
- 设置每图变体数。
- 设置输出命名规则。
- 只显示异常项。
- 清空无效项。

确认前校验：

- 至少有 1 张有效待处理图片。
- 至少有 1 条有效任务。
- 每条启用任务必须有待处理图片引用。
- 每条启用任务必须有非空 prompt，或有可展开成 prompt 的操作预设。
- 当前模型必须支持图片编辑。
- 每条任务的图片数量不得超过当前模型和 provider 的限制。
- 如果存在缺失文件引用、无效图片、空 prompt 或模型不兼容，确认按钮必须禁用或要求修复。

## 11. 执行与队列

确认后，系统应把批量改图预览展开为现有任务队列可执行的图片编辑任务：

- `mode` 使用 `edit`。
- 每条任务的 `imageFiles` 来自该条的 `inputImageRefs` 和 `sharedReferenceRefs`。
- 第 1 张图片默认是待处理目标图；后续图片是共享参考图；如有 mask，则由支持 mask 的 provider adapter 单独传递。
- 共享参数继承当前表单快照，包括模型、尺寸、质量、输出格式、背景、moderation、provider 和 providerOptions。
- per-item overrides 只允许覆盖安全字段，并在确认前经过模型能力校验。
- 批量入队必须批量 setState，避免逐条高频更新造成主线程抖动。
- 保留现有并发控制、取消、重试、错误、流式限制和历史写入逻辑。

任务元数据至少应增加：

- `batchId`
- `batchIndex`
- `batchTotal`
- `batchLabel`
- `batchInputImageId`
- `batchInputImageFilename`
- `batchInputImageRelativePath`
- `batchInputImageOrder`
- `batchVariantIndex`
- `batchVariantTotal`

## 12. 结果、历史与导出

批量改图结果必须能让用户快速确认“哪张原图对应哪张结果”：

- 任务卡片和历史项显示原始文件名或缩略图。
- 同一批次结果可以按输入图片分组。
- 单个失败项可以重试，不影响其他图片。
- 批量级操作至少支持重试失败项和清空失败项；后续可扩展导出成功项。
- 历史写入保留输入图文件名、相对路径、输入图序号、变体序号和 batch 信息。
- 如果同一输入图生成多张结果，需要标记输出序号，例如 `image_001_variant_02`。

建议输出命名模板：

```text
{originalName}_{operation}_{batchIndex}
{relativeDir}/{originalName}_{variantIndex}
{batchLabel}_{inputIndex}_{variantIndex}
```

P0 可以只把命名提示写入历史和下载默认名；P1 再做批量打包下载或批次导出。

## 13. 草稿与恢复

批量改图草稿必须独立于普通 prompt draft 和现有 batch plan draft，或在现有草稿结构中显式标记 `source: image-edit-batch`。

草稿应保存：

- 批量改图模式。
- 共享 prompt 或操作预设。
- 待处理图片摘要：文件名、相对路径、数量、大小、缩略图可选摘要。
- 共享参考图摘要。
- 当前表单快照。
- 最近一次预览计划。
- 用户手工修改和锁定状态。

草稿不得保存：

- 大图片 Blob/base64 到 localStorage。
- API Key、密码、分享密钥或 Base URL 凭证。
- Web 端完整本地绝对路径。
- 未经用户授权可直接读取的任意本地文件路径。

刷新页面后：

- Web 端如无法恢复 File 对象，应展示“需要重新选择图片”的可恢复草稿，而不是静默提交缺失图片。
- Tauri 端如支持安全本地引用恢复，也必须在执行前检查文件仍存在且未变为目录或无权限文件。

## 14. Web 与 Tauri 要求

Web：

- 多文件选择、拖拽文件和支持能力内的文件夹拖入在浏览器端完成。
- 图片文件只在用户选择或拖入后进入内存引用；不通过 JSON 远程读取本地文件。
- 执行仍使用现有 direct / proxy Web 路径。

Tauri：

- 文件夹选择、拖入文件夹、文件读取和桌面代理执行必须通过 `src/lib/desktop-runtime.ts` 暴露的封装或对应 Rust command。
- 桌面端不能依赖 Next.js API route 才能完成本地批量改图。
- 本地文件路径不得进入云同步或分享链接。

共同要求：

- 当前模型不支持图片编辑时，批量改图模式不可确认。
- 当前 provider 对单次图片数量有限制时，按每条任务校验，而不是按整批图片总数误判。
- 分享链接禁用批量时，批量改图入口同样受控。

## 15. UI 与 i18n 要求

- 所有新增可见文案必须进入 `src/lib/i18n/*`，中英文等支持语言保持同步。
- 不使用 `window.alert`、`window.prompt` 或 `window.confirm`。
- 批量待处理图片列表在移动端可滚动，不遮挡确认按钮。
- 缩略图、文件名、路径和错误 badge 在浅色和深色主题下都要可读。
- 长文件名和长路径必须截断显示，并保留 tooltip 或展开查看能力。
- 批量改图面板应保持工具型界面，不做营销式 hero 或装饰性区块。
- 关键动作不依赖 hover，触屏设备可完成上传、选择、删除、确认和重试。

## 16. 性能与安全

- 拖入文件夹后应分批枚举和校验图片，不能一次性同步处理大量文件导致界面卡死。
- 缩略图使用对象 URL 或轻量预览缓存，移除图片后及时释放。
- 不对所有图片做全量字节 hash；重复检测使用轻量元数据即可。
- 大批量执行继续遵守现有并发上限和取消机制。
- 单条任务只在即将执行时读取必要文件内容，避免提前把整批图片都转成 base64。
- JSON 导入中的本地路径、远程 URL、API Key、Base URL、password、connectionMode 等字段必须忽略并提示。
- 文件名、相对路径和 metadata 只作为文本展示，不进行 HTML 渲染。

## 17. 配置要求

批量配置建议新增以下选项：

| 配置项               | 默认建议                      | 说明                         |
| -------------------- | ----------------------------- | ---------------------------- |
| 批量改图默认模式     | 统一指令逐图处理              | 空 prompt 打开时的默认落点   |
| 单批最大待处理图片数 | 50                            | 防止误拖入超大目录           |
| 大批量确认阈值       | 20                            | 超过后确认前提醒             |
| 每图默认变体数       | 1                             | P1 可暴露                    |
| 允许文件夹递归导入   | 开启                          | Web 不支持时自动降级         |
| 默认输出命名模板     | `{originalName}_{batchIndex}` | 用于历史和下载名             |
| 默认操作预设         | 自定义                        | 可切换老照片修复、风格迁移等 |

配置迁移要求：

- 旧配置不存在批量改图配置时补齐默认值。
- 不从提示词润色配置隐式继承批量改图行为。
- 不把批量待处理图片草稿写入云同步配置。

## 18. 验收标准

1. 主 prompt 为空时，批量按钮仍可点击打开批量面板。
2. 主 prompt 非空时，批量按钮显示高亮可用状态，并把当前 prompt 带入批量面板。
3. 用户可以在批量面板独立上传多张待处理图片，不影响主表单源图片区。
4. 用户可以在支持的桌面环境拖入文件夹，系统递归识别图片并显示相对路径。
5. 用户上传 10 张图片并输入一条风格迁移 prompt 后，可以生成 10 条图片编辑预览任务。
6. 用户选择老照片修复预设时，即使主 prompt 为空，也能生成有效批量改图预览。
7. 当前源图片区有 1 张参考图、批量待处理区有 5 张图片时，系统可以创建 5 条“目标图 + 共享参考图”的编辑任务。
8. 如果模型单次图片数不足以容纳目标图和参考图，预览区会提示切换模型或减少参考图，不能静默创建失败任务。
9. 确认入队后，每条任务保留原始文件名、输入图序号、批次序号和变体序号。
10. 单条任务失败时，其他任务继续执行，失败项可单独重试。
11. 历史结果能看出每张输出对应哪张输入图。
12. 刷新页面后，Web 端不会静默提交丢失文件的草稿；需要重新选择图片时有明确提示。
13. Web 与 Tauri 桌面端都能完成多图片批量改图；文件夹能力不支持时有明确降级说明。
14. 所有新增 UI 文案已覆盖当前支持语言，浅色/深色和移动/桌面布局无溢出。

## 19. 分期建议

### 19.1 P0

- 批量按钮空 prompt 可打开，按是否有 prompt 区分普通状态和高亮状态。
- 批量面板新增独立待处理图片区。
- 支持多文件选择、文件拖拽和桌面能力内的文件夹拖入。
- 支持统一指令逐图处理。
- 支持内置操作预设：风格迁移、老照片修复、图片增强、自定义。
- 支持当前源图片作为共享参考图。
- 预览项展示缩略图、文件名、prompt、参考图状态、启用状态和输出命名提示。
- 确认后展开为图片编辑任务入队。
- 历史和任务元数据记录输入图文件名、输入图序号和 batch 信息。
- Web 与 Tauri 基础路径可用，文件夹不支持时降级明确。

### 19.2 P1

- 每图多变体。
- JSON 导入按文件名、相对路径或 externalId 匹配图片。
- 批量输出命名模板设置。
- 批量结果按输入图分组查看。
- 前后对比视图。
- 批量导出成功结果。
- 批量图片列表虚拟化。
- 批量配置页增加图片数量上限、默认模式和输出命名模板。

### 19.3 P2

- 蒙版批量编辑。
- 目标图、参考图、mask 的复杂配对。
- 基于图生文为每张图片自动生成 per-item prompt。
- 批量多阶段工作流，例如先修复再风格化。
- 支持表格 / CSV 导入并映射到 per-item prompt。
- 支持更细的 provider 图片角色映射。

## 20. 开放问题

- P0 单批默认最大图片数应定为 30、50 还是更高，需要结合移动端和桌面端性能测试决定。
- 共享参考图是否只复用当前源图片区，还是在批量面板内再提供独立参考图区。
- 当模型不支持图片角色时，是否统一约定“第一张为目标图，后续为参考图”，还是按 provider 分别限制。
- 批量输出打包下载是否进入 P1，还是作为 P0 验收的一部分。
- 老照片修复等操作预设是否允许用户在批量配置中自定义默认 prompt。
