# OpenAI 多模态图生文与闭环规划（GPT-5.5 / GPT-5.4）

## 文档状态

- **目的**：给后续的 OpenAI 多模态能力接入定一个清晰边界，先解决“图生文”，再评估“图生图编排”。
- **适用范围**：GPT Image Playground 的 OpenAI / OpenAI-compatible 接口。
- **时间基准**：以下判断基于 2026-05-12 可见的 OpenAI 官方文档。

## 结论先行

1. **GPT-5.5 / GPT-5.4 更适合做“图生文”层**。它们是多模态理解模型，核心价值是接收图片输入，再输出可用于生成的文本。
2. **不要把它们塞进现有图像生成模型列表**。当前 `IMAGE_MODELS` 这一层是“图像生成/编辑模型”语义，参数和能力都不同。
3. **图像生成主链路继续保留 `gpt-image-2`**。这条链路已经有尺寸、质量、输出格式、编辑、蒙版等成熟能力，适合作为默认出图引擎。
4. **第二阶段再做 GPT-5.5 / GPT-5.4 的图生图编排**。如果要让它们参与“直接出图”，应该走 Responses API 的工具编排，而不是把它们当成传统图片模型。

## 官方依据

- OpenAI 模型页说明，最新模型支持图片输入，且是文本输出为主。
- `gpt-5.5` / `gpt-5.4` 的模型页和图像视觉指南都明确支持图片输入，并区分了 `detail` 级别。
- 模型页的工具列表里也有 `Image generation`，说明多模态模型可以进入工具编排式出图流程，但这仍然不是现有 `gpt-image-2` 的直接替代。
- 图像生成 API 当前仍以 `gpt-image-2` 这类图像模型为中心。

参考：
- https://developers.openai.com/api/docs/models
- https://developers.openai.com/api/docs/models/gpt-5.5
- https://developers.openai.com/api/docs/models/gpt-5.4
- https://developers.openai.com/api/docs/guides/images-vision
- https://developers.openai.com/api/reference/resources/images/methods/generate

## 目标闭环

用户理想路径应该是：

1. 上传一张图片。
2. 选择一个多模态理解模型，例如 `gpt-5.5` 或 `gpt-5.4`。
3. 生成一份可直接用于文生图的提示词。
4. 一键把提示词回填到主提示词框。
5. 切回图像生成模型，例如 `gpt-image-2`。
6. 直接生成，再继续编辑或优化。

这条链路的重点不是“让模型替用户看图说话”这么简单，而是要把输出变成可复用、可编辑、可直接再生成的提示词资产。

## 能力分层

### 1. 图像生成层

职责：

- 接收 prompt。
- 接收尺寸、质量、输出格式、背景、审核、压缩等生成参数。
- 返回图片结果。

当前适合保留的模型：

- `gpt-image-2`
- 现有兼容图像模型

### 2. 图生文层

职责：

- 接收一张或多张图片。
- 识别主体、构图、光照、材质、风格、文本元素、负面约束。
- 输出结构化提示词，而不是纯闲聊。

建议接入模型：

- `gpt-5.5`
- `gpt-5.4`

建议默认输出：

- 主提示词
- 负向提示词
- 风格标签
- 构图与镜头说明
- 适合的长宽比建议

### 3. 图生图编排层

职责：

- 让多模态模型先理解图片，再决定是否调用图像生成工具。
- 更适合做实验功能、智能助手式流程或自动化工作流。

建议放到第二阶段：

- 通过 Responses API 的 `image_generation` tool 做编排。
- 不替换现有 `gpt-image-2` 默认出图链路。

## 现有代码应怎么接

当前仓库里，图像模型注册表和高级参数面板都已经按“图像生成模型”设计了，所以这次不要直接扩展 `src/lib/model-registry.ts` 里的图像模型集合去装 `gpt-5.5` / `gpt-5.4`。

更合理的做法是新增一层：

- `VisionModelDefinition` 或类似命名。
- `supportsImageInput`
- `supportsStructuredOutput`
- `supportsResponsesApi`
- `supportsImageGenerationTool`
- `preferredDetail`

这样做的好处是：

- 图像生成模型和多模态理解模型不会互相污染。
- 供应商能力差异可以单独判断。
- 未来加别的 Vision 模型时，不需要改坏现有图像生成面板。

## 推荐交互

### 首版入口

建议至少放 3 个入口：

- 历史卡片里的“图生文”。
- 全屏预览里的“生成提示词”。
- 编辑区源图块里的“提取提示词”。

### 首版输出

结果页不要只给一段长文本，建议同时给：

- 一段可直接回填的主提示词。
- 一个可折叠的结构化结果区。
- 一个“复制为提示词”按钮。
- 一个“发送到生成器”按钮。

### 细节选项

`detail` 建议做成高级项，默认自动：

| 模型 | 默认建议 | 说明 |
| --- | --- | --- |
| `gpt-5.5` | `auto` | 官方文档说明 `auto` 和默认省略值等同于 `original`，适合高细节图、UI 截图、海报、信息图。 |
| `gpt-5.4` | `auto` | 官方文档说明 `auto` 会按 `high` 档位处理，适合常规图像理解；密图可显式切到 `original`。 |

如果后面要给用户暴露得更细，可以再开放 `low` / `high` / `original` 的明确选择。

## 请求与返回设计

### 请求

输入图片建议支持：

- 单图。
- 多图参考。
- 本地文件上传。
- 拖拽和粘贴。

实现时要统一成同一种后端输入格式，不要让前端分别拼两套逻辑。

### 返回

建议用结构化输出，尽量不要只靠自由文本。

推荐字段：

- `prompt`
- `negative_prompt`
- `style_tags`
- `subject`
- `composition`
- `lighting`
- `color_palette`
- `aspect_ratio`
- `notes`

如果目标端点不支持结构化输出，再降级成纯文本，但要保留稳定的解析格式。

## 技术落点

### 后端

建议新增独立服务，而不是直接复用现有文生图 route：

- `src/app/api/image-to-prompt/route.ts`
- 或者等价的 vision route

如果桌面端要走 Tauri 代理，也建议新增独立 command，例如：

- `proxy_image_to_prompt`

### 配置

建议给图生文单独一组配置，但默认可复用 OpenAI 配置：

- API Key
- Base URL
- 模型 ID
- `detail`
- 是否启用结构化输出

这样可以做到：

- 没配专用配置时，自动复用 OpenAI Key 和 Base URL。
- 配了专用配置时，单独走图生文模型。

### 兼容性

OpenAI 官方接口和 OpenAI-compatible 接口不要一刀切。

需要按能力判断：

- 是否支持图片输入。
- 是否支持结构化输出。
- 是否支持 Responses API。
- 是否支持 `image_generation` tool。

没有这些能力的端点，直接在 UI 上隐藏对应入口，别让用户点进去才报错。

## 安全与体验边界

1. **不要默认自动上传图片**。必须用户显式点击。
2. **不要把原图静默同步到云端**。默认只保存最终提示词和必要的元数据。
3. **不要在结果里编造真实身份信息**。人物、品牌、地点、文字内容必须只基于可见信息或用户明确提示。
4. **大图要有保护**。如果输入图太大，先做尺寸或体积判断，再决定是否压缩或降采样。
5. **成本要可见**。图生文比普通提示词润色贵，UI 上最好给一个明确的动作按钮，不要做成隐式自动行为。

## 测试计划

### 单元测试

- 请求体组装。
- `detail` 默认值和归一化。
- 多图输入顺序。
- 结构化输出解析。
- OpenAI-compatible 与官方 OpenAI 的分支选择。

### 集成测试

- 官方 OpenAI 端点。
- 一个兼容端点。
- 桌面代理链路。

### 人工验收

至少测这几类图：

- 人像照片。
- 商品图。
- 海报 / KV。
- UI 截图。
- 信息图 / 图表。
- 多图参考融合。

验收重点：

- 提示词是否足够可生成。
- 是否保留了画面主体和限制条件。
- 是否能一键回填到生成器。
- 是否能顺利回到 `gpt-image-2` 出图。

## 第二阶段

第二阶段才考虑 GPT-5.5 / GPT-5.4 的“图生图编排”：

- 通过 Responses API 做多模态编排。
- 用 `image_generation` tool 让模型直接驱动出图。
- 作为实验性模式提供，不替代默认图像生成链路。

原因很简单：

- 这条链路更复杂。
- 兼容性更依赖端点能力。
- 成本和调试成本都更高。
- 现有 `gpt-image-2` 已经足够稳定地承担默认出图。

## 结论

最稳的落地顺序是：

1. 先做 GPT-5.5 / GPT-5.4 的图生文。
2. 用结构化输出把提示词变成资产。
3. 再把这条链路接回现有 `gpt-image-2` 生成流。
4. 最后再评估 Responses API 的 `image_generation` tool 编排。

这样能确保“文生图 -> 图生文 -> 再文生图”的闭环真的可用，而不是只多了一个不能落地的演示按钮。
