# API Key 购买入口与后台配置管理需求文档

## 1. 背景

首次使用缺配置引导已经把专业错误收敛为：

```text
请先配置服务供应商端点和填写 API Key。 点我配置
```

这解决了“用户不知道去哪里配置”的问题，但还没有解决更早一步的问题：很多小白用户连 API Key 都没有，也不知道应该去哪个供应商站点申请、充值或购买。

如果用户点击 **点我配置** 后看到 API Key 输入框，但自己没有 Key，仍然会卡住。此时最短闭环应该是：

```text
请先配置服务供应商端点和填写 API Key。 点我配置 购买 API Key
```

其中 **购买 API Key** 不应该写死在代码里，因为不同部署方可能推荐不同供应商、代理服务、套餐页面或帮助文档。按钮文本和跳转 URL 必须由管理后台配置；后台未配置或配置无效时，前台完全不显示该按钮，避免出现不可用入口。

同时，后续还会有更多站点级配置从 `.env` 迁移到管理后台。例如展示内容、短链策略、公开引导链接、桌面远程服务、站点开关、公共提示文案、服务端 fallback 配置等。如果每个功能都单独做一套配置表和读取接口，会让后台、公共 API、Web/Tauri 读取逻辑和部署文档越来越分散。因此本需求需要同时规划一个可扩展的后台配置管理模型。

## 2. 目标

1. 在所有“缺服务供应商端点 / 缺 API Key / 缺任务模型”的快捷引导中，支持追加一个由后台配置的纯文字按钮。
2. 该按钮默认建议文案为“购买 API Key”，但最终显示文本必须来自后台配置。
3. 点击按钮后打开后台配置的供应商站点 URL。
4. 后台未配置、未启用、URL 无效、当前公共运行时无法读取配置时，前台不显示该按钮。
5. 管理后台支持维护多个 API Key 购买入口，并允许管理员按需启用其中一个作为当前公开入口。
6. 公开读取接口只返回前台需要的安全字段，不暴露后台备注、审计信息、管理员信息或任何 secret。
7. Web、Tauri 桌面和 Tauri Android 的缺配置引导语义一致；如果某个运行时无法读取远程后台配置，应安全降级为只显示 **点我配置**。
8. 为未来后台统一管理 `.env` 相关配置预留通用配置模型、优先级规则、权限、审计和安全边界。

## 3. 非目标

- 不在本需求中实现自动开户、代购、支付、优惠码核销、余额查询或供应商账号绑定。
- 不自动获取、保存或代填用户的 API Key。
- 不把部署方配置的购买链接当成官方背书。它只是部署方提供的外部入口。
- 不改变现有供应商端点配置流程。用户购买或申请 Key 后，仍需回到应用中填写端点和 API Key。
- 不把真实供应商错误改成购买引导。无效 Key、余额不足、限流、权限不足、模型不存在等仍应展示对应错误。
- 不要求第一阶段完成所有 `.env` 配置后台化。第一阶段只落地 API Key 购买入口，并给后续配置中心规划边界。

## 4. 用户体验需求

### 4.1 展示位置

当缺配置引导出现时，如果公开配置中存在可用购买入口，提示区展示：

```text
请先配置服务供应商端点和填写 API Key。 点我配置 购买 API Key
```

要求：

1. **点我配置** 仍然排在前面，负责把用户带到正确的应用内配置界面。
2. **购买 API Key** 排在后面，负责把没有 Key 的用户带到外部供应商站点。
3. 两个按钮都使用纯文字按钮样式，不能做成大面积填充按钮，避免抢占结果区注意力。
4. 移动端空间不足时允许按钮换行，但文本不能溢出、重叠或遮挡。
5. 错误提示的关闭按钮仍保留，用户可以手动关闭整块错误提示。
6. 如果后台未配置购买入口，则 UI 与当前版本保持一致，只显示 **点我配置**。

### 4.2 适用场景

购买入口应覆盖所有已有缺配置引导位置：

| 场景 | 当前已有动作 | 新增动作 |
| --- | --- | --- |
| 生图 / 图片编辑结果区错误 | 点我配置 | 购买 API Key |
| 图生文 / 多模态结果区错误 | 点我配置 | 购买 API Key |
| 提示词润色错误提示 | 点我配置 | 购买 API Key |
| 批量 AI 规划错误提示 | 点我配置 | 购买 API Key |
| 视频生成缺配置提示 | 点我配置 | 购买 API Key |
| 任务追踪器中的缺配置错误 | 点我配置 | 购买 API Key |
| 设置页内无可用端点空状态 | 添加端点 / 点我配置 | 可选显示购买 API Key |

要求：

- 不要只在结果区加按钮，任务追踪器、批量弹窗、润色错误等同类缺配置提示也要统一。
- 非缺配置错误不显示购买入口。例如 `invalid_api_key` 应提示 Key 无效，不能只引导购买。
- 配置跳转和购买跳转互不影响。点击购买不会自动关闭设置弹窗或清空当前任务输入。

### 4.3 跳转行为

Web 端：

1. 点击购买按钮后使用新标签页打开外部 URL。
2. 链接必须带 `rel="noopener noreferrer"`。
3. 不允许在当前页面直接跳转导致用户丢失当前输入。

Tauri 桌面端：

1. 使用 `openExternalUrl` 打开系统浏览器。
2. 不直接在 WebView 内加载外部供应商页面。
3. 打开失败时显示短错误通知，不影响当前页面状态。

Tauri Android：

1. 优先使用系统外部浏览能力。
2. 如果运行时不支持外部打开，则复制链接或提示用户在浏览器打开，不能让应用崩溃。

### 4.4 文案

按钮文本由后台配置，示例：

- `购买 API Key`
- `申请 API Key`
- `获取 API Key`
- `去供应商购买`
- `购买中转额度`

前台不得硬编码固定可见文案作为最终显示文本。可以在后台新建表单中提供默认建议值，但保存后前台必须使用后台值。

文本要求：

1. 中文建议 2 到 12 个字符，英文建议 2 到 32 个字符。
2. 不能包含换行、HTML、Markdown、脚本片段。
3. 前台渲染为纯文本，不使用 `dangerouslySetInnerHTML`。
4. 多语言 V1 可以先只配置一个通用 label；V2 支持 `labelByLocale`。

## 5. 后台管理需求

### 5.1 后台入口

建议在管理后台新增一个顶层或二级入口：

```text
后台 -> 站点配置 -> API Key 购买入口
```

如果后续统一配置中心同步推进，也可以命名为：

```text
后台 -> 站点配置 -> 公开引导
```

第一阶段不要把它放进普通用户的系统设置里，因为这是部署管理员对整个站点的公共配置，不是单个用户的本地偏好。

### 5.2 列表能力

后台列表展示所有购买入口配置项：

| 字段 | 说明 |
| --- | --- |
| 名称 | 后台识别用，不一定公开展示 |
| 按钮文本 | 前台按钮显示文本 |
| URL | 外部供应商或购买页面 |
| 状态 | 启用 / 停用 |
| 当前公开 | 当前公开使用的唯一项 |
| 更新时间 | 最近修改时间 |
| 操作 | 编辑、启用为当前项、停用、删除、打开预览 |

要求：

1. 可以创建多个配置项。
2. 同一时间最多只有一个配置项作为公开购买入口。
3. 启用某项为当前公开入口时，自动取消其他项的当前公开状态。
4. 停用当前公开入口后，前台不显示购买按钮，除非管理员启用另一项。
5. 删除当前公开入口前需要二次确认。
6. 列表支持按名称、按钮文本、URL host 搜索。

### 5.3 表单字段

V1 推荐字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | string | 自动 | 稳定 ID |
| `name` | string | 是 | 后台识别名称 |
| `buttonLabel` | string | 是 | 前台按钮文本 |
| `url` | string | 是 | 点击后打开的外部 URL |
| `enabled` | boolean | 是 | 是否启用该项 |
| `active` | boolean | 是 | 是否作为当前公开入口 |
| `description` | string | 否 | 后台备注，不下发前台 |
| `sortOrder` | number | 否 | 未来支持多入口时使用 |
| `createdAt` | datetime | 自动 | 创建时间 |
| `updatedAt` | datetime | 自动 | 更新时间 |
| `updatedBy` | string | 自动 | 最近修改管理员 |

V2 预留字段：

| 字段 | 说明 |
| --- | --- |
| `labelByLocale` | 多语言按钮文案，例如 `zh-CN`、`en-US` |
| `providerHints` | 适用供应商，例如 OpenAI、Gemini、Anthropic、OpenAI 兼容中转 |
| `taskScopes` | 适用任务，例如 image、visionText、polish、batch、video |
| `domainConstraints` | 指定域名下使用不同购买入口 |
| `startsAt` / `endsAt` | 活动时间窗 |
| `priority` | 多入口选择优先级 |

V1 只要求全站唯一公开入口。V2 再考虑“图片缺配置显示图片供应商购买入口，图生文显示文本模型供应商购买入口”等更细分策略。

### 5.4 校验规则

按钮文本：

1. trim 后不能为空。
2. 不允许换行。
3. 不允许 HTML 标签。
4. 长度超限时阻止保存。

URL：

1. 必须是绝对 URL。
2. 生产环境只允许 `https://`。
3. 开发环境可以允许 `http://localhost` 用于本地测试，但不能下发给公网部署。
4. 不允许 `javascript:`、`data:`、`file:`、`blob:`、`chrome:` 等协议。
5. 不允许 userinfo，例如 `https://user:pass@example.com`。
6. 不允许 localhost、私网 IP、链路本地地址或保留地址作为公网公开配置。
7. 保存前显示归一化后的 host，方便管理员确认。

如果项目已有 URL 安全校验工具，应复用或扩展现有工具，不要新增宽松的 ad hoc 字符串判断。

### 5.5 权限与审计

1. 只有后台管理员可以创建、编辑、启用、停用或删除购买入口。
2. 所有变更写入审计日志，至少记录：
    - 操作类型。
    - 配置项 ID 和名称。
    - 修改前后摘要。
    - 操作管理员。
    - 时间。
3. 审计日志不记录任何 secret。虽然购买入口本身不应包含 secret，但仍要避免把未来配置中心的敏感字段错误写入明文日志。
4. 后台接口必须复用现有管理员认证、CSRF / Origin 防护和请求来源校验。

## 6. 公共读取接口

### 6.1 推荐接口

建议新增或扩展一个公开运行时配置接口：

```http
GET /api/public-runtime-config
```

返回示例：

```json
{
  "apiKeyPurchaseCta": {
    "label": "购买 API Key",
    "url": "https://supplier.example.com/buy"
  }
}
```

未配置时：

```json
{
  "apiKeyPurchaseCta": null
}
```

要求：

1. 接口不需要登录即可读取，因为按钮本来就是公开展示。
2. 接口只能返回前台展示必要字段：`label`、`url`。
3. 不返回后台 `name`、`description`、`updatedBy`、审计信息、内部 ID。
4. 不返回任何 API Key、服务端 secret、管理员配置原文。
5. 如果后台配置非法或已停用，返回 `null`。
6. 读取失败时前台静默降级为不显示购买入口，不要额外打扰用户。

### 6.2 缓存策略

建议：

1. 服务端可以对公开配置做短 TTL 缓存，例如 30 到 120 秒。
2. 前端可以在应用启动时读取一次，并在失败后退避重试。
3. 后台保存后不要求所有已打开页面立即刷新，但新打开页面应尽快生效。
4. 如果已有展示内容公共配置接口，需评估是否合并，避免首屏发起过多公共配置请求。

### 6.3 Tauri 与静态导出

桌面端静态导出不能依赖本地 Next API route。因此 Tauri 路径需要明确配置来源：

1. Web 部署：读取当前站点的 `/api/public-runtime-config`。
2. Tauri 桌面：优先读取桌面端配置的远程服务地址；如果未配置远程服务或请求失败，则隐藏购买按钮。
3. Tauri Android：同桌面端，必须请求远程公共配置服务，不能假设本地 API route 存在。
4. 如果后续把公开配置合并到展示内容服务，桌面端应复用现有桌面展示服务读取方式。

## 7. 前台集成要求

### 7.1 数据读取

建议新增前端运行时配置 hook 或 store：

```typescript
type PublicRuntimeConfig = {
    apiKeyPurchaseCta: null | {
        label: string;
        url: string;
    };
};
```

要求：

1. 应用启动后异步读取，不阻塞首屏渲染。
2. 读取失败时值为 `null`。
3. 所有缺配置提示共享同一个读取结果。
4. 不在每个错误组件里重复请求接口。
5. URL 在前端使用前仍进行一次轻量校验，避免后端异常或旧缓存导致危险链接渲染。

### 7.2 UI 组件抽象

建议把缺配置提示抽象成共享组件或共享渲染函数，避免各处重复实现：

```typescript
type ConfigurationRequiredActionsProps = {
    onConfigure: () => void;
    purchaseCta?: {
        label: string;
        url: string;
    } | null;
};
```

组件输出：

```text
请先配置服务供应商端点和填写 API Key。 点我配置 [后台配置的购买按钮]
```

要求：

1. **点我配置** 使用现有任务上下文目标。
2. 购买按钮只负责外部打开，不尝试推断用户要买哪个模型或哪个套餐。
3. 组件必须支持在结果区 Alert、任务追踪器、批量弹窗、润色错误区域等不同容器中使用。
4. 组件不能把后台配置 label 当作 HTML。

### 7.3 行为细节

1. 点击购买按钮后，当前错误提示仍保持显示，方便用户购买后回来继续点 **点我配置**。
2. 如果用户手动关闭错误提示，购买按钮随错误提示一起消失。
3. 如果后台配置在用户打开页面后变更，当前页面不要求实时更新；刷新或下一次公共配置轮询后生效。
4. 如果 URL 打开失败，显示短通知，例如“无法打开外部链接”，不改变配置状态。

## 8. 与 `.env` 后台配置化的关系

### 8.1 配置分层

后续统一后台配置中心需要区分三类配置：

| 类型 | 示例 | 是否公开下发 | 是否可含 secret |
| --- | --- | --- | --- |
| 公开 UI 配置 | API Key 购买入口、展示位开关、公共提示文案 | 可以 | 不允许 |
| 服务端运行配置 | 短链策略、允许来源、图片存储默认模式、公共服务 URL | 部分字段可以 | 通常不允许 |
| 服务端 secret 配置 | 部署级 OpenAI Key、S3 fallback secret、Webhook secret | 不公开 | 可以，但必须加密和脱敏 |

API Key 购买入口属于 **公开 UI 配置**，是统一配置中心的第一批低风险落地点。

### 8.2 优先级规则

建议未来统一规则：

```text
后台运行时配置 > 环境变量 > 代码默认值
```

但需要区分敏感级别：

1. 公开 UI 配置可以由后台直接覆盖 `.env`，保存后短时间内生效。
2. 服务端运行配置可以后台覆盖，但需要权限、审计和回滚能力。
3. secret 配置第一阶段仍建议保留 `.env`，后台只显示“已由环境变量配置”状态，不显示原值。
4. 如果后续支持后台保存 secret，必须使用加密存储、脱敏展示、只写不读、轮换审计和备份恢复策略。

### 8.3 配置定义模型

建议建立统一配置定义注册表：

```typescript
type AdminConfigDefinition<TValue> = {
    key: string;
    title: string;
    description: string;
    category: 'public-ui' | 'server-runtime' | 'server-secret';
    valueType: 'string' | 'number' | 'boolean' | 'url' | 'json' | 'secret';
    envName?: string;
    defaultValue: TValue;
    publicExposure: 'never' | 'safe-public' | 'status-only';
    sensitive: boolean;
    validator: (value: unknown) => TValue;
};
```

API Key 购买入口可以作为结构化配置，不一定放入简单 key-value 表，但应接入同一套：

- 分类导航。
- 校验。
- 审计。
- 公共读取映射。
- 环境变量 fallback。
- 导出和备份策略。

### 8.4 与现有后台能力融合

当前后台已有展示内容、展示组、短链等管理能力。统一配置中心应避免重复造管理框架：

1. 复用现有管理员认证。
2. 复用现有后台数据库路径和迁移体系。
3. 复用审计日志能力。
4. 复用请求来源和可信 origin 校验。
5. 复用后台列表、表单、保存成功/失败通知组件。
6. 与展示内容公共读取服务协调，避免 Web 和 Tauri 分别维护两套公开配置读取逻辑。

## 9. 数据模型建议

### 9.1 表结构

如果继续使用后台 SQLite，建议新增表：

```sql
CREATE TABLE admin_public_action_configs (
    id TEXT PRIMARY KEY,
    kind TEXT NOT NULL,
    name TEXT NOT NULL,
    button_label TEXT NOT NULL,
    target_url TEXT NOT NULL,
    enabled INTEGER NOT NULL DEFAULT 0,
    active INTEGER NOT NULL DEFAULT 0,
    description TEXT,
    sort_order INTEGER NOT NULL DEFAULT 0,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    updated_by TEXT
);
```

`kind` V1 固定为：

```text
api_key_purchase
```

约束：

1. 同一 `kind` 最多一个 `active = 1 AND enabled = 1` 的记录。
2. `button_label` 和 `target_url` 保存前必须通过服务端校验。
3. `description` 不下发前台。

如果数据库不方便做部分唯一索引，可以在服务端事务中保证启用某项时先取消同类其他项。

### 9.2 公共 DTO

```typescript
type PublicApiKeyPurchaseCta = {
    label: string;
    url: string;
};
```

公共接口不返回 ID，避免前台把它当作稳定业务标识依赖。

### 9.3 未来多入口模型

V2 如果需要按任务、供应商或域名展示不同购买入口，可以扩展为：

```typescript
type PublicApiKeyPurchaseCtaRule = {
    label: string;
    url: string;
    taskScopes: Array<'image' | 'visionText' | 'polish' | 'batch' | 'video'>;
    providerHints: string[];
    priority: number;
};
```

前台选择规则：

1. 当前任务精确命中优先。
2. 当前供应商提示命中其次。
3. 全局默认兜底。
4. 仍然只展示一个购买按钮，避免小白用户看到多个外链。

V1 不做复杂选择，只返回一个全局公开入口。

## 10. 安全与合规要求

1. 购买入口 URL 是外部链接，必须当作潜在钓鱼面处理。
2. 管理员保存时必须看到目标 host。
3. 前台按钮应具备可访问名称；如果按钮文本来自后台，aria label 可以使用同文本或追加 host，但不要暴露过长 URL。
4. 公共接口不得被用作开放重定向。接口只返回 JSON，前台自己打开链接；不要提供 `/api/redirect?url=...`。
5. 后台不允许保存危险协议或私网地址。
6. 外部打开必须使用 noopener，防止供应商页面反向控制原页面。
7. 公开配置读取失败不能阻断生图、设置或历史功能。
8. 按钮文本和 URL 不写入用户分享链接，不进入同步快照，不随工作空间迁移。
9. 如果部署方配置的是商业购买入口，文档应提醒部署方自行承担供应商合规、收费、退款、隐私和数据处理说明责任。

## 11. 兼容性与迁移

1. 旧部署没有后台配置时，前台不显示购买按钮。
2. 旧后台数据库没有新表时，迁移后创建空表，不生成默认购买入口。
3. 不从 `.env` 自动生成购买链接，避免管理员无意公开旧测试链接。
4. 如果后续增加 `API_KEY_PURCHASE_URL` 之类环境变量，也只作为后台配置中心的 fallback，不应优先于后台显式配置。
5. 分享链接、云同步、本地配置导入导出不包含购买入口配置，因为这是站点级公共配置。

## 12. 验收标准

### 12.1 前台

1. 后台未配置购买入口时，缺配置提示只显示 **点我配置**。
2. 后台配置并启用购买入口后，缺配置提示显示 **点我配置** 和后台配置的购买按钮。
3. 点击 **点我配置** 仍按当前任务进入最短配置入口。
4. 点击购买按钮在新标签页或系统浏览器打开后台配置 URL。
5. 生图、图生文、提示词润色、批量 AI 规划、视频和任务追踪器中的缺配置提示都能显示购买按钮。
6. 非缺配置错误不显示购买按钮。
7. 移动端和桌面端文字不溢出，关闭错误提示后按钮一起消失。
8. 公开配置读取失败时不显示购买按钮，也不额外报错。

### 12.2 后台

1. 管理员可以新增、编辑、停用、删除购买入口。
2. 管理员可以从多个入口中启用一个作为当前公开入口。
3. 启用一个入口后，其他入口不再公开。
4. URL 协议、私网地址、危险协议、空 label 等非法配置无法保存。
5. 保存、启用、停用、删除都有审计记录。
6. 公共读取接口只返回启用且当前公开的入口，并只包含 label 和 URL。

### 12.3 配置中心规划

1. 新增文档明确 `.env`、后台配置、公开读取和 secret 的分层。
2. 后续新增后台配置时有统一 key、类型、校验、公开暴露和审计模型可遵循。
3. 不把用户本地供应商 API Key 和站点级部署配置混为一类。

## 13. 测试要求

### 13.1 单元测试

- API Key 购买配置归一化。
- URL 校验。
- 单 active 选择规则。
- 公共 DTO 脱敏。
- 缺配置引导组件在有 / 无购买配置时的渲染。
- 非缺配置错误不渲染购买按钮。

### 13.2 API 测试

- 管理员 CRUD 权限。
- 未登录不能修改。
- 公共读取未配置返回 `null`。
- 公共读取只返回启用且 active 的配置。
- 非法 URL 保存失败。
- 启用新项会取消旧 active。

### 13.3 浏览器测试

- 320px 移动宽度下错误提示、点我配置、购买按钮和关闭按钮不重叠。
- 浅色和深色主题都可读。
- 点击购买按钮打开新标签页。
- 点击关闭按钮隐藏整块结果区错误。
- 点击点我配置仍打开正确配置入口。

### 13.4 Tauri 测试

- 桌面端读取远程公共配置成功时显示购买按钮。
- 远程公共配置不可用时隐藏购买按钮。
- 点击购买按钮调用外部浏览器打开。
- WebView 不内嵌加载供应商站点。

## 14. 实施阶段建议

### Phase 1: API Key 购买入口闭环

1. 新增后台数据表和迁移。
2. 新增后台购买入口管理页。
3. 新增公共运行时配置接口。
4. 前台缺配置引导读取公共配置并展示购买按钮。
5. 覆盖 Web 端主要缺配置入口。
6. 补充文档和测试。

### Phase 2: 跨运行时与组件收敛

1. 将缺配置动作抽成共享组件。
2. 覆盖任务追踪器、批量弹窗、润色面板、设置空状态等所有位置。
3. Tauri 桌面和 Android 读取远程公共配置。
4. 外部打开失败时统一通知。

### Phase 3: 后台统一配置中心

1. 建立配置定义注册表。
2. 将现有公开 UI 类 `.env` 配置纳入后台。
3. 将部分服务端非 secret 配置纳入后台。

5. 建立配置导出、备份、审计和回滚策略。
