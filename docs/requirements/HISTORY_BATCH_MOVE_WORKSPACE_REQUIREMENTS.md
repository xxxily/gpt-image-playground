# 生成历史批量移动工作空间需求文档

## 1. 背景与问题

生成历史进入多选模式后，当前已支持批量下载、批量删除和批量同步等操作。工作空间能力上线后，历史记录已经具备 `workspaceId` 和 `workspaceNameSnapshot` 归属字段，但用户仍缺少一个低成本的批量整理入口。

典型场景：

1. 用户先在默认工作空间里快速试验，后续希望把一批满意结果移动到某个项目工作空间。
2. 用户在 **全部工作空间** 视图里检索到跨项目结果，希望统一归档到新的主题空间。
3. 用户误把历史生成在错误工作空间，需要批量修正归属，而不是逐条重新生成或删除。
4. 移动端多选工具条空间有限，下载、删除、同步、移动等主操作需要优先展示；当前 **取消** 和 **退出多选** 语义重复，窄屏下应隐藏低优先级的 **取消**。

因此需要在生成历史多选工具条新增 **移动** 操作，允许用户把选中的历史记录及其关联文件归属移动到目标工作空间。

## 2. 目标

1. 图片生成/编辑历史进入多选后，工具条新增 **移动** 操作。
2. 支持把选中历史批量移动到另一个创作工作空间。
3. 点击 **移动** 后弹出工作空间选择器，用户选择目标工作空间并确认移动。
4. 如果当前只有一个可用工作空间，选择器展示引导空状态，提醒用户先创建工作空间，并提供 **新建工作空间** 按钮。
5. 用户点击 **新建工作空间** 后，复用现有工作空间新建面板或 Dialog；创建成功后可继续执行移动。
6. 移动操作更新历史记录的工作空间归属，并同步更新相关文件所有权元数据；不做不必要的 Blob 复制或重传。
7. 移动后历史区按当前 scope 刷新：如果当前在源工作空间视图，已移动走的条目从列表中消失；如果在 **全部工作空间** 视图，条目保留并更新工作空间徽标。
8. 移动端多选工具条在空间不足时隐藏 **取消** 按钮，保留下载、移动、同步、删除等实质操作；用户仍可通过顶部 **退出多选** 退出。

## 3. 非目标

- 不实现跨账号、跨设备实时协作或多人权限迁移。
- 不把移动等同于复制；首版只改变历史和文件归属，不保留源工作空间副本。
- 不因为移动历史而修改供应商配置、模型配置、提示词模板、云存储配置或批量配置。
- 不为移动操作请求远端删除权限。移动不是删除，默认不删除远端对象。
- 不引入原生 `window.alert`、`window.confirm` 或 `window.prompt`。
- 不强制用户先创建多个工作空间；只有触发移动且没有可选目标时才引导创建。

## 4. 范围

### 4.1 P0 范围

P0 覆盖图片生成/编辑历史多选：

- `HistoryMetadata[]`
- 输出图片引用
- 源图引用，如果该历史类型保存了源图
- 工作空间徽标和当前历史 scope 刷新
- 手动云同步的元数据变更

### 4.2 P1 扩展

图生文历史和视频历史也具备工作空间归属，建议在同一套批量移动能力稳定后扩展：

- `VisionTextHistoryMetadata[]`
- `VideoHistoryMetadata[]`
- 对应源图、输出视频、本地文件指针和同步状态

如果实现时历史面板已经统一了三类历史的多选工具条，允许一次性覆盖三类；但不能跨标签混选，仍按当前历史标签独立操作。

## 5. 用户流程

### 5.1 正常移动流程

1. 用户在历史区点击 **多选**。
2. 用户勾选一条或多条历史。
3. 多选工具条显示已选数量，并展示 **下载**、**移动**、**同步**、**删除** 等操作。
4. 用户点击 **移动**。
5. 系统打开 **移动到工作空间** 选择器。
6. 用户选择目标工作空间。
7. 用户点击 **移动 N 项**。
8. 系统更新选中历史的 `workspaceId` 和 `workspaceNameSnapshot`。
9. 系统清空当前选择并退出或保留多选模式；建议 P0 成功后退出多选，降低连续误操作风险。
10. 系统展示成功 Notice，例如 **已移动 12 项到「客户 A」**。

### 5.2 只有一个工作空间时

当可用目标工作空间数量为 0 时，也就是除当前唯一工作空间外没有其他 active 工作空间：

1. 用户点击 **移动**。
2. 系统打开工作空间选择器，但不展示空白列表。
3. 选择器展示说明：当前只有一个工作空间，需先创建新工作空间才能移动历史。
4. 选择器提供 **新建工作空间** 主按钮。
5. 点击后打开现有 **新建工作空间** 面板或 Dialog。
6. 创建成功后：
   - 推荐行为：返回移动选择器，并自动选中新建工作空间，用户再点击 **移动 N 项** 确认。
   - 可选快捷：创建 Dialog 底部提供 **创建并移动 N 项**，但必须明确文案，避免用户误解为只创建空间。

### 5.3 全部工作空间视图

在 **全部工作空间** 视图中：

- 允许选择来自多个源工作空间的历史并统一移动到一个目标工作空间。
- 目标列表中应显示所有 active 工作空间。
- 如果某些选中项已经属于目标工作空间，确认前提示 **其中 X 项已在目标工作空间，将保持不变**。
- 移动成功后条目仍保留在列表里，但工作空间徽标应立即更新。

### 5.4 当前工作空间视图

在具体工作空间视图中：

- 目标列表默认排除当前工作空间，避免无意义移动。
- 如果用户通过搜索或调试状态选中了当前工作空间，确认按钮应禁用并说明 **已在当前工作空间**。
- 移动成功后，被移动条目从当前列表移除。
- 如果当前页所有条目都被移走，显示该工作空间历史空状态。

## 6. 工作空间选择器

选择器建议使用项目内 Dialog 或移动端全屏 Drawer，复用现有主题、表单和 i18n 能力。

核心内容：

- 标题：**移动到工作空间**
- 描述：展示选中数量，例如 **将 12 项历史移动到所选工作空间。**
- 搜索框：按工作空间名称、描述筛选。
- 工作空间列表：
  - 名称
  - 颜色 / 图标
  - 描述
  - 历史数量
  - 当前工作空间标记
  - 已归档状态
- 底部按钮：
  - **取消**
  - **新建工作空间**
  - **移动 N 项**

目标工作空间规则：

- active 工作空间默认可选。
- archived 工作空间默认不展示；如果展示，需要放在折叠区并明确标识，P0 不建议允许移动到 archived。
- 默认工作空间可以作为目标，只要它不是当前唯一空间且不是当前选中项的唯一源空间。
- 工作空间名称过长时必须截断，不能挤压确认按钮。

## 7. 新建工作空间联动

批量移动不应复制一套新的工作空间创建表单，应复用现有 `CreativeWorkspacesPanel` / `CreativeWorkspacesDrawer` 中的新建逻辑或抽出共享 Dialog。

联动要求：

1. 新建表单校验继续使用现有规则：trim 后不能为空、不能重名、不能使用保留名称。
2. 新建成功后，工作空间列表立即刷新。
3. 新建成功后不应丢失当前选中的历史集合。
4. 如果新建失败，错误展示在新建表单内，不关闭移动选择器。
5. 新建成功后可以自动选中新工作空间，但最终移动仍需要用户点击明确确认按钮。

## 8. 数据与同步语义

### 8.1 历史归属更新

移动本质是更新历史元数据：

```ts
type WorkspaceScopedMetadata = {
    workspaceId?: string;
    workspaceNameSnapshot?: string;
};
```

执行移动时：

- `workspaceId` 写为目标工作空间 id。
- `workspaceNameSnapshot` 写为目标工作空间当前展示名。
- `updatedAt` 或对应历史更新时间字段如存在，应更新为当前时间。
- 历史排序默认仍按原始生成时间，不因为移动而置顶；后续可增加 **最近移动** 筛选。

### 8.2 文件归属更新

移动应覆盖用户理解中的“文件移动过去”：

- 选中历史关联的输出图片、源图、图生文源图、视频输出等文件指针，应归属到目标工作空间。
- 如果底层只是 `HistoryMetadata.images[]` 文件名引用，优先通过历史条目的 `workspaceId` 解释归属，不扫描 Blob 内容。
- 如果存在 `WorkspaceFilePointer` 或资源索引，需要把相关指针的 `workspaceId` 一并更新。
- 不复制 IndexedDB Blob、不重新上传远端对象、不重命名本地物理文件，除非后续文件系统设计明确要求按工作空间目录隔离。
- 被多个历史共同引用的文件不应被重复移动或破坏引用；共享引用需要保留 `shared` 或引用计数语义。

### 8.3 云同步

移动后的同步语义：

- 移动会让轻量配置和历史元数据变为待同步。
- 普通 **同步配置** 或 **同步历史** 应把新的 `workspaceId` 上传到 manifest / snapshot。
- 移动不产生远端删除 tombstone。
- 移动不要求远端对象 `DeleteObject` 权限。
- 远端恢复时必须尊重移动后的工作空间归属。
- 如果另一台设备仍有旧归属，后续冲突策略应以现有 manifest 时间戳、device id 或冲突 UI 为准，不在本需求单独定义协作合并。

## 9. 交互细节

### 9.1 工具条按钮优先级

多选工具条操作优先级建议：

1. 已选数量
2. 下载
3. 移动
4. 同步
5. 删除
6. 取消

其中 **取消** 和顶部 **退出多选** 语义重复。桌面端空间足够时可以保留 **取消**，移动端或窄容器下一律隐藏 **取消**，确保下载、移动、同步、删除正常显示。

### 9.2 移动端收敛规则

移动端建议按容器宽度使用响应式规则：

- `>= 420px`：可显示 **下载 / 移动 / 同步 / 删除 / 取消**。
- `< 420px`：隐藏 **取消**，保留 **下载 / 移动 / 同步 / 删除**。
- `< 360px`：优先保留 **移动 / 删除**，下载和同步可缩为图标按钮或放入 **更多** 菜单。
- 顶部仍保留 **退出多选**，作为唯一退出入口。

按钮不能换行挤压到两行导致历史卡片跳动；可使用横向滚动、图标按钮或 **更多** 菜单，但 P0 推荐先隐藏重复的 **取消**。

### 9.3 反馈与错误

- 未选择条目时，**移动** 按钮禁用。
- 没有可用目标时，展示新建引导，而不是禁用后无反馈。
- 部分条目移动失败时，保留失败选择并展示失败数量。
- 全部失败时不清空选择。
- 成功后展示 Notice，并提供短时间 **撤销** 可作为 P1；P0 可以不做撤销，但失败信息必须明确。

## 10. 可访问性与国际化

- 所有新增文案必须进入 `src/lib/i18n/*`。
- **移动** 按钮需要可理解的 `aria-label`，例如 **移动选中的历史到工作空间**。
- 工作空间列表使用单选语义，键盘可用方向键浏览、Enter 确认选择。
- Dialog 打开后初始焦点放在搜索框或第一个可选工作空间。
- 移动成功、失败、无目标工作空间状态应通过 Notice 或 Dialog 内文本表达，不能只依赖颜色。
- 不使用原生浏览器 modal API。

## 11. 推荐实现边界

建议涉及模块：

| 模块 | 责任 |
| --- | --- |
| `src/components/history-panel.tsx` | 多选工具条新增移动按钮，连接移动 Dialog |
| `src/components/history/history-batch-move-dialog.tsx` | 工作空间选择器和只有一个工作空间时的新建引导 |
| `src/components/workspaces/*` | 复用或抽出新建工作空间 Dialog |
| `src/lib/creative-workspace-history.ts` | 增加批量改写 workspace scope 的纯函数 |
| `src/lib/image-history.ts` | 持久化图片历史批量移动 |
| `src/lib/vision-text-history.ts` | P1：图生文历史批量移动 |
| `src/lib/video-history.ts` | P1：视频历史批量移动 |
| `src/lib/sync/*` | 确认 snapshot 包含移动后的归属 |
| `src/lib/i18n/*` | 新增文案和移动端收敛相关 label |

纯函数建议：

```ts
type MoveHistoryToWorkspaceResult<T> = {
    moved: T[];
    unchanged: T[];
    missingIds: string[];
};

function moveHistoryEntriesToWorkspace<T extends WorkspaceScopedMetadata>(
    entries: readonly T[],
    selectedIds: readonly string[],
    target: WorkspaceTaskScope
): MoveHistoryToWorkspaceResult<T>;
```

实际实现可按图片历史当前使用的 timestamp / id 选择 id 类型，但应避免在组件内散落数组改写逻辑。

## 12. 验收标准

1. 图片历史多选后可看到 **移动** 操作。
2. 选择两条图片历史并移动到另一个工作空间后，历史记录的 `workspaceId` 和 `workspaceNameSnapshot` 更新。
3. 在源工作空间视图中移动成功后，已移动条目不再显示。
4. 在 **全部工作空间** 视图中移动成功后，条目仍显示，但工作空间徽标更新为目标工作空间。
5. 只有默认工作空间时点击 **移动**，展示创建工作空间引导和 **新建工作空间** 按钮。
6. 从移动选择器中新建工作空间后，原选中历史集合不丢失，并可继续移动。
7. 移动不删除本地文件，不请求远端删除权限，不触发 Blob 全量扫描。
8. 移动后的历史参与下一次同步和恢复，恢复后仍归属目标工作空间。
9. 移动端窄屏多选工具条不展示重复的 **取消** 按钮，其他关键操作不被挤出或重叠。
10. 新增文案在所有支持语言中都有翻译。
11. 浅色和深色主题下 Dialog、按钮、空状态和 Notice 均可读。

## 13. 测试建议

- `creative-workspace-history` 纯函数测试：
  - legacy 历史没有 `workspaceId` 时可从默认工作空间移动。
  - 选中项移动到新工作空间。
  - 目标与当前归属相同的条目保持不变。
  - 缺失 id 返回 `missingIds`。
- 历史持久化测试：
  - 批量移动后 localStorage / IndexedDB 写入新归属。
  - 移动失败时不破坏原历史数组。
- UI 测试：
  - 多选工具条显示移动按钮。
  - 只有一个工作空间时显示新建引导。
  - 创建工作空间后仍能继续移动。
  - 移动端宽度下隐藏 **取消** 且按钮不重叠。
- 同步测试：
  - snapshot 中历史归属为移动后的目标。
  - 移动不生成远端删除 tombstone。

## 14. 分期建议

### Phase 1：图片历史批量移动

- 图片历史多选工具条新增 **移动**。
- 工作空间选择器和新建引导。
- 图片历史归属批量更新。
- 移动端隐藏重复 **取消**。
- i18n 和基础测试。

### Phase 2：图生文与视频历史

- 图生文历史多选支持移动。
- 视频历史多选支持移动。
- 源图、输出视频和文件指针归属更新。

### Phase 3：增强整理能力

- 移动成功后的撤销。
- 批量复制到工作空间。
- 按工作空间批量筛选、最近移动筛选。
- 更多菜单承载极窄屏低频操作。
