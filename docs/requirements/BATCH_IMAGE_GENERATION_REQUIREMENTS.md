# GPT Image Playground 批量生图与任务规划需求文档

## 1. 背景与现状

当前项目已经具备一套完整的单任务生图工作台：

- `src/components/editing-form.tsx` 负责提示词输入、历史、模板、润色、图生文和高级参数。
- `src/lib/prompt-polish.ts` / `src/lib/prompt-polish-core.ts` 负责把一段输入润色成一条高质量提示词。
- `src/hooks/useTaskManager.ts` 负责任务排队、执行、取消、重试和历史回写。
- `src/components/task-tracker.tsx` 和 `src/components/task-card.tsx` 负责任务队列与单任务结果展示。
- `src/components/image-output.tsx` / `src/components/text-output.tsx` 负责右侧结果区切换。

这些能力足够支撑“单条 prompt -> 单个任务”，但还没有“把一段长文本、一个创作需求，或一组参考图 + 文案自动规划成多个任务，并在提交前逐条预览、编辑、确认”的能力。

批量场景的痛点很明确：

- 用户输入的是多条文案、分段脚本、完整文章或长说明，不知道该拆成几个任务。
- 用户输入的也可能只是一个需求，例如“围绕这张参考图做 8 个海报方向”，系统需要从需求中自动生成多个候选版本。
- 用户可能提供一组源图片作为参考图，再输入一段文案，希望一次性围绕参考图批量产出多个设计结果，从中挑选最满意的一版。
- 用户希望先让 AI 规划，再人工修正，而不是一次性直接提交。
- 批量任务如果直接展开，会污染现有单条提示词历史，也会把扁平任务列表撑得很难用。

补充说明：不依赖 AI 的批量文本切分、特殊分隔符切分和外部 JSON 导入需求，详见 [批量任务手动切分与 JSON 导入需求文档](./BATCH_TASK_MANUAL_SPLIT_AND_JSON_IMPORT_REQUIREMENTS.md)。

## 2. 目标

1. 用户输入一段文本、一个创作需求，或添加源图片后输入文案，可以点击 `批量` 进入批量规划流程。
2. `AI 自动` 是默认选项，由 AI 判断当前输入更适合“拆分内容”还是“生成多版本候选”，并自动建议任务数量。
3. 系统仍支持用户手动指定任务数量；固定数量是高级选项，不是默认入口。
4. 批量规划完成后，右侧结果区显示任务预览，不直接创建真实任务。
5. 用户可以继续通过输入提示词调整整批方案，也可以单独编辑每条任务的 prompt。
6. 用户确认后，预览中的每条任务才真正进入现有任务队列。
7. 生成后的子任务仍沿用现有的队列、输出、错误、重试、历史和同步体系。
8. 首次实现必须支持带源图片的批量任务：当前源图片会作为整批任务的参考输入，被每个子任务继承。
9. Web 和 Tauri 都要可用，不引入 Node-only 或桌面专属路径。

## 3. 非目标

- 首版只做批量生图 / 批量编辑图片任务，不做批量视频、批量图生文。
- 首版支持“当前源图片整体继承到每个子任务”，不要求实现每个子任务选择不同参考图组合。
- 首版不做复杂的多图参考权重、局部蒙版批量变体或参考图自动分组。
- 不要求输入时自动后台拆分，必须由用户显式点击批量。
- 不把批量规划结果直接塞进现有单条 prompt history，不允许无区分地污染历史。
- 不要求首版实现复杂的批次模板市场、批次分享或多轮 Agent 工作流。

## 4. 与现有代码的关系

### 4.1 可以复用的现有结构

- `editing-form.tsx` 的 prompt 工具栏已经有润色、历史、模板、图生文等按钮，批量入口应放在同一组工具里。
- `prompt-polish` 的 preset + temporary custom prompt + direct/proxy/Tauri 路由，是最接近批量规划的交互和传输模板。
- `useTaskManager.ts` 已经具备 `queued/running/streaming/done/error/cancelled` 状态和并发控制。
- `TaskTracker` / `TaskCard` 已经能处理单任务结果、错误和重试，后续只需要补 batch group。
- `ImageOutput` / `TextOutput` 的“右侧结果区切换”逻辑可以保留，但批量预览需要单独的列表型输出面板。

### 4.2 当前缺口

- 现有提示词历史只记录单条 prompt，不适合直接塞入 N 条子任务。
- 现有任务列表是扁平结构，无法表达一组批次任务的整体状态。
- 现有结果区是“单任务结果态”，不是“可编辑预览态”。

## 5. 用户流程

1. 用户在提示词框输入原始文本。
2. 用户可选添加源图片。若已有源图片，批量流程默认把这些图片作为整批参考图。
3. 点击 `批量`。
4. 打开批量规划面板，默认带入当前输入文本与当前源图片上下文。
5. 批量目标默认是 `AI 自动`。用户可切换到：
    - 拆分内容
    - 多版本探索
    - 固定数量
6. 用户填写或选择批量规划提示词。
7. 点击 `生成预览`。
8. 右侧结果区显示批量预览列表。
9. 用户可：
    - 继续调整整批方案
    - 逐条编辑子任务 prompt
    - 禁用、删除、重排某条任务
    - 重新生成预览
10. 点击 `确认创建批量任务`。
11. 系统将有效条目展开为多个普通生图任务，按顺序入队。
12. 任务跑完后，继续沿用现有结果区和历史行为。

## 6. 批量规划要求

### 6.1 默认策略

- `AI 自动` 必须是默认选中项。
- AI 自动不是简单“自动拆几条”，而是先判断用户意图：
    - 多条文案 / 文章 / 脚本：优先拆分成多个主题明确的任务。
    - 单个需求 / 同一段文案：优先生成多个风格、构图、场景、受众或版式方向的候选任务。
    - 源图片 + 文案：优先围绕参考图做多版本产出，保持参考图关键主体或视觉约束。
- AI 自动必须返回选择该策略的原因，便于用户理解和继续调整。

### 6.2 内容拆分原则

- 保留原始意图，不改变主线主题。
- 优先按段落、条目、章节、场景、产品点、卖点、镜头段落切分，而不是机械按字数平切。
- 多条文案尽量一条文案对应一条任务。
- 长文应优先生成“可直接用于出图”的独立提示词，而不是把原文拆碎后照搬。
- 初始顺序应与源文本语义顺序一致。
- 不确定、缺失或冲突信息要进入 warning，不要静默编造。

### 6.3 多版本探索原则

- 当用户输入的是同一个需求或同一段文案时，批量工具应生成多个差异明确的版本，而不是多条同质 prompt。
- 版本差异可以来自：
    - 视觉风格：写实摄影、插画、3D、极简、商业海报等。
    - 构图版式：居中主体、对角线构图、信息图、分屏、留白等。
    - 场景设定：室内、户外、产品棚拍、社媒封面、电商详情等。
    - 受众与语气：高端商务、年轻潮流、科技感、温暖生活方式等。
    - 镜头和光线：近景、广角、顶光、逆光、柔光、强对比等。
- 每个候选版本必须说明它的探索方向，帮助用户快速比较。
- 如果用户要求“多出几版看看”，AI 应优先多样化，不要只改同义词。

### 6.4 参考图批量原则

- 如果当前存在源图片，批量工具必须把源图片纳入规划上下文。
- 首版默认策略：所有启用的子任务继承同一组源图片。
- 子任务 prompt 需要明确如何使用参考图，例如“保留主体轮廓”“沿用配色和构图”“基于参考图做海报化重设计”。
- 若当前模型不支持图片编辑或参考图输入，预览面板必须提示用户切换到支持图片编辑的模型，不能静默创建会失败的任务。
- 当源图片只是风格或构图参考，prompt 应避免要求模型逐像素复制；当用户明确要求保留主体时，应把保留要求写入每条子任务。
- 批量执行时，带源图片的任务应走现有 `image-edit` 路径；无源图片时走 `image-generate` 路径。

### 6.5 数量策略

| 模式     | 行为                                      |
| -------- | ----------------------------------------- |
| AI 自动  | AI 自行判断任务数，用户可以给最大数量约束 |
| 固定数量 | 用户指定准确数量，AI 必须尽量满足         |

要求：

- `AI 自动` 是默认模式。
- `AI 自动` 模式只暴露最大任务数约束；`固定数量` 模式只暴露目标任务数，避免两个数字同时出现造成关系不清。
- `AI 自动` 模式下，结果必须返回 `recommendedCount` 和置信提示。
- 若源文本过短，不应为了凑数强拆。
- 若源文本过长，可以拆多条，但要避免碎片化；若输入是单个需求，则应优先做多版本探索。
- 若 AI 返回的数量超出上限，必须在预览里明确提示。

### 6.6 输出结构

建议批量规划返回结构化 plan，而不是一条长文本：

```ts
type BatchPlan = {
    batchId: string;
    sourceText: string;
    sourceImageCount: number;
    planningMode: 'auto' | 'content-split' | 'variant-exploration' | 'reference-variant' | 'mixed';
    resolvedIntent: 'content-split' | 'variant-exploration' | 'reference-variant' | 'mixed';
    countMode: 'fixed' | 'auto';
    targetCount?: number;
    recommendedCount: number;
    summary: string;
    strategyReason: string;
    warnings: string[];
    tasks: BatchPlanItem[];
};

type BatchPlanItem = {
    id: string;
    order: number;
    enabled: boolean;
    title?: string;
    sourceExcerpt: string;
    variationAxis?: string;
    prompt: string;
    negativePrompt?: string;
    notes?: string;
    sourceImagePolicy?: 'inherit-all' | 'none';
    lockedByUser?: boolean;
};
```

说明：

- `prompt` 是最终进入生图任务的核心字段。
- `negativePrompt` 可选，不要求每条都填。
- `sourceExcerpt` 用于帮助用户确认这一条是从哪一段拆出来的。
- `variationAxis` 用于解释这一条和其他候选版本的差异。
- `sourceImagePolicy` 首版只需要支持 `inherit-all`，表示继承当前源图片区的全部图片。
- `lockedByUser` 用于保护人工修改过的条目，避免后续 AI 调整时被覆盖。

### 6.7 失败与修复

- 规划结果必须经过结构校验。
- 如果 AI 返回的不是 JSON，先展示原始输出，再提供修复或重新生成。
- 如果只有部分条目合法，应允许先保留合法条目，再继续补全其他条目。
- 当 `tasks` 数组因为缺少逗号、尾逗号或截断导致整体 JSON 解析失败时，客户端应尽量提取已完整返回的合法任务对象，不能把整批预览直接判失败。
- 单条失败不能导致整批全部失效。

## 7. 结果预览与编辑

### 7.1 预览面板

- 右侧结果区在批量模式下显示列表型预览，不展示单张图片结果。
- 预览项至少包含：
    - 序号
    - 一句话摘要
    - 可编辑 prompt
    - 可选 negative prompt
    - 探索方向或拆分依据
    - 来源片段
    - 参考图继承状态
    - 启用 / 禁用状态
    - 是否已被用户修改
- 预览区域必须可滚动，不能挤压出界。
- 长 prompt 要支持折叠、展开、复制。

### 7.2 继续调整

- 预览面板顶部保留一个“继续调整”输入框。
- 用户可对当前整批方案下达二次指令，例如“把前两条改成海报风格”“基于参考图多做几版科技感”或“减少商业化措辞”。
- AI 二次调整时应尽量保留手工编辑过的条目。
- 如果二次调整会覆盖人工修改，必须显式提示并要求确认。

### 7.3 手工编辑

- 每条子任务必须支持单独编辑 prompt。
- 可选支持单条禁用、删除、复制、上移、下移、重置为 AI 建议。
- 未启用条目不会被创建为真实任务。
- 空 prompt 条目必须在确认前拦截或要求补全。

### 7.4 确认前检查

- 确认按钮应显示实际将创建的任务数量。
- 如果数量超过安全上限，应要求二次确认。
- 如果存在无效、重复或被截断内容，应给出明确提示。

## 8. 批量任务创建与队列

### 8.1 创建方式

- 确认后，预览项逐条展开为 `useTaskManager` 可接收的普通任务。
- 每个子任务继承当前生图模型、尺寸、质量、输出格式、背景、moderation、provider 等共享参数。
- 如果当前有源图片，子任务还必须继承当前源图片，并按现有逻辑进入 `image-edit` 任务路径。
- 如果当前没有源图片，子任务进入 `image-generate` 任务路径。
- 子任务的差异主要来自 prompt、variationAxis 和必要的条目级覆盖字段。
- 创建过程应是批量 enqueue，而不是逐条高频 setState 抖动主线程。

### 8.2 任务分组

- 所有子任务应共享一个 `batchId`。
- 任务状态中应保留 `batchIndex`、`batchTotal`、`batchLabel` 等分组信息。
- `TaskTracker` 应能按 batch 展示整体进度，并允许展开单条任务。
- 单条任务的取消、重试、查看结果仍要可用。
- 任务栏应提供批量级动作：重试失败项、清空失败项（界面按钮显示为“清空”）；后续可继续扩展全部取消、展开 / 收起。

### 8.3 历史写入

- 每个子任务仍写入现有图片历史。
- 建议给 `HistoryMetadata` 增加可选 batch 字段，至少保留 `batchId` 和 `batchIndex`。
- 旧历史必须兼容，新增字段只能是可选。

## 9. 草稿与恢复

- 批量规划草稿应独立于普通 prompt draft。
- 草稿至少保存：
    - 原始输入文本
    - 源图片引用摘要，例如数量和文件名，不保存 Blob/base64 到 localStorage
    - 批量目标
    - 规划模式
    - 目标数量 / 最大数量
    - 规划提示词
    - 最近一次预览结果
- 打开批量面板时若存在未提交草稿，应显示恢复横条。
- 草稿恢复后，用户应能继续编辑而不丢当前输入。
- 草稿保存必须做体积保护，超限时只保留轻量字段，不要强行塞大 JSON。
- Batch identity 应使用生成的 `batchId`，不要在热路径上对整段长文本做重型 hash。

## 10. 与润色的关系

- 批量规划应复用润色的整体壳：
    - 相同的 API Key / Base URL / 模型选择逻辑
    - 相同的直连 / 代理 / Tauri 路径
    - 相同的 preset 搜索与临时自定义 prompt 交互
- 但批量规划不是润色字符串，输出契约必须是结构化 plan。
- 建议把能力名独立成 `prompt.batchPlan` 或等价名称，避免和 `prompt.polish` 混淆。
- 默认系统提示词应明确要求“只输出可解析的批量计划 JSON”，并优先使用 `AI 自动` 判断内容拆分或多版本探索。

## 11. 运行时与 i18n

- 所有新增可见文案必须进入 i18n。
- 不能只在 Web 生效，Tauri 桌面端同样要能规划和确认批量任务。
- 不能依赖 Node-only 服务器能力。
- 需要沿用现有 direct / proxy / desktop command 处理方式。
- 批量面板在桌面可用宽面板，在移动端应切成可滚动、可收起的布局。
- 长文本、长列表或按钮文字都不能溢出容器。

## 12. 验收标准

1. 输入一段多段文章后，点击批量可以生成可编辑的任务预览。
2. 输入一个创作需求或同一段文案后，批量工具可以自动生成多个差异明确的候选版本。
3. `AI 自动` 是批量面板默认选项，用户可以改为固定数量。
4. 用户可以逐条修改、禁用、删除或重排子任务。
5. 用户可以继续输入调整指令并重新生成预览。
6. 用户添加源图片后，批量工具会围绕当前参考图和文案生成多个可执行的批量任务。
7. 确认后，批量任务会被展开为多个普通生图 / 图片编辑任务并进入现有队列。
8. 批量任务的执行结果仍可进入现有历史和结果区。
9. Web 与 Tauri 都能工作，刷新或误关页签后可恢复草稿。
10. 所有新文案均已纳入 i18n。
