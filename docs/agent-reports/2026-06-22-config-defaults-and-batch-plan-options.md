## 问题与解决

| 问题                                                                                                                             | 解决办法                                                                                                                                                            | 剩余风险                                                                                |
| -------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| `normalizeUnifiedProviderModelConfig()` 将显式默认模型当作派生结果重新筛选，旧配置中缺 task 或不在 endpoint allowlist 时被删除。 | 新增显式默认模型收集和修复逻辑：只针对 `vision.text`、`prompt.polish`、`prompt.batchPlan`，在兼容端点上补 capability，并补齐 endpoint allowlist。                   | 如果 endpoint 被删除、禁用，或 catalog entry 完全不存在，默认模型仍按安全规则失效。     |
| 同一个文本模型同时作为润色和批量规划默认模型时，单值补 task 可能只保留一个任务。                                                 | `explicitDefaultTasks` 改为 entry -> task set，一次补齐多个缺失 task。                                                                                              | 无已知剩余风险。                                                                        |
| 设置页端点下拉变化直接 `delete` 默认 task，用户未确认新模型就保存会丢旧默认模型。                                                | 移除三处删除逻辑；图生文端点下拉使用非持久浏览状态，确认选择模型后才更新 `selectedVisionTextProviderInstanceId` 和 `visionTextModelId`。                            | 用户如果删除端点或禁用模型，默认模型仍会失效，这是预期清理。                            |
| 批量预览生成时的 `formSnapshot` 被确认提交路径优先使用，导致预览后修改画质、尺寸、输出格式等不生效。                             | `EditingFormHandle` 暴露 `getBatchFormSnapshot()`；Workbench 的预览生成、确认提交和兼容性检查都调用当前快照，旧草稿快照只作为无法读取当前表单时的 fallback helper。 | 未新增完整 E2E 自动化断言 `submitTasks` payload；已用单元测试和浏览器场景覆盖主要风险。 |
