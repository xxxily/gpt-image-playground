## 问题与解决

| 问题                                          | 解决办法                                                                                                                                                                                                     | 剩余风险                                                                           |
| --------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------- |
| 当前工作区存在发布前已有的无关文档改动        | 本次只暂存 CORS 修复提交和发布相关文件；无关文件保持未提交状态。                                                                                                                                             | `git status` 会继续显示这些无关文件，不代表发布提交包含它们                        |
| 原发布规范默认包含 `142` 与 `129` 两台服务器  | 按用户最新要求，将标准发布流程改成仅部署 `129`，`142` 只在明确恢复或补发时执行。                                                                                                                             | `142` 域名不会随 `2.15.3` 自动更新；恢复后需要按 tag 单独补发                      |
| 发布前生产依赖审计失败                        | `@tailwindcss/vite` 属于构建期工具但位于生产依赖，导致 `vite` / `esbuild` 被 `npm audit --omit=dev` 计入生产链路；本次将其移到 `devDependencies`，并用 npm overrides 锁定 `vite@6.4.3` 与 `esbuild@0.28.1`。 | npm 全量审计仍有 1 条 dev 中危提示；发布门 `audit:prod` 已恢复为 0 vulnerabilities |
| 根 Vitest 收集独立任务服务的 `node:test` 文件 | 根 `vitest.config.ts` 排除 `gpt-image-task-service/**`，独立服务继续通过 `npm test --prefix gpt-image-task-service` 使用自己的 Node test runner 验证。                                                       | 后续新增独立子项目时，也要明确根测试和子项目测试边界                               |
| 首次 `v2.15.3` Release workflow typecheck 失败 | CI 根环境只安装当前 App 依赖，但根 `tsconfig` 仍收集独立任务服务源码，导致缺少子项目 Hatchet SDK 类型；本次将 `gpt-image-task-service` 从当前 App 的 `tsconfig.json` 和 `tsconfig.typecheck.json` 中排除。 | 需要删除并重发 `v2.15.3` tag，让 tag 指向修正后的 release commit |
| 发布报告需要记录 Actions 和线上部署证据       | 先随 release commit 提交报告骨架，发布完成后再回填最终证据并单独提交。                                                                                                                                       | 报告最终提交会位于 `v2.15.3` tag 之后                                              |
