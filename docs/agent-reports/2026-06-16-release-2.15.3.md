## 问题与解决

| 问题 | 解决办法 | 剩余风险 |
| --- | --- | --- |
| 当前工作区存在发布前已有的无关文档改动 | 本次只暂存和提交发布相关文件；无关文档改动保持未提交状态。 | `git status` 会继续显示这些无关文件，不代表发布提交包含它们 |
| 原发布规范默认包含 `142` 与 `129` 两台服务器 | 按用户最新要求，将标准发布流程改成仅部署 `129`，`142` 只在明确恢复或补发时执行。 | `142` 域名不会随 `2.15.3` 自动更新；恢复后需要按 tag 单独补发 |
| 发布前生产依赖审计失败 | 将 `@tailwindcss/vite` 移到 `devDependencies`，并用 npm overrides 锁定 `vite@6.4.3` 与 `esbuild@0.28.1`。 | npm 全量审计仍有 1 条 dev 中危提示；发布门 `audit:prod` 已恢复为 0 vulnerabilities |
| 首次 `v2.15.3` Release workflow typecheck 失败 | 将 `gpt-image-task-service` 从当前 App 的 `tsconfig.json` 和 `tsconfig.typecheck.json` 中排除；独立服务继续使用自己的测试命令验证。 | 后续新增独立子项目时，也要明确根检查和子项目检查边界 |
| 第二次 Release workflow Windows `npm ci` 失败 | Windows runner 从浮动 `windows-latest` 固定到 `windows-2022`，避免新版 VS 18 被当前 `node-gyp@10` 识别失败；同时让 Release workflow 在构建完成前保持 draft。 | 后续 GitHub runner 环境继续变化时，仍需关注 native 依赖和 Node/npm 版本兼容 |
| 本地 `.env.production` 缺少生产 auth secret | 部署 `129` 时从远程共享环境读取已有 `BETTER_AUTH_SECRET`，通过子进程环境变量传入 `scripts/deploy-129.sh`；日志和报告不记录密钥值。 | 后续发版仍需避免空值覆盖远端认证配置 |
| Outline 操作日志同步失败 | 已按 129 规范写入本地操作日志；检查 `~/.env.for.skill` 后重试 Outline CLI，Outline API 仍返回 HTTP 502。 | 需要 Outline 服务恢复后补同步本地操作日志到“操作记录”分类 |
| GitHub Actions annotations 包含 Node 20 deprecation 与 checkout git 128 warning | 逐 job 核对 conclusion，最终 run 所有 job 均为 `success`；这些 annotations 未导致构建失败。 | GitHub 将继续推动 Node 24，后续可评估 actions 版本和 runner 环境 |
| macOS 未配置 Apple 签名/公证 secrets | workflow 按发布规范上传 unsigned/unnotarized DMG，并在 Release notes 中包含 Gatekeeper 用户侧说明。 | macOS 用户首次打开可能需要按说明解除 quarantine |
