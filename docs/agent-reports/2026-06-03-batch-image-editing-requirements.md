## 问题与解决

| 问题                                                              | 解决办法                                                                                              | 剩余风险                                                                |
| ----------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| 批量改图预览保存到草稿后会被普通 `BatchPlan` 归一化，导致映射丢失 | 新增 `normalizeBatchImageEditPlan()`，草稿恢复时保留 `taskFamily`、`imageInputs` 和每条任务输入图映射 | Web 刷新后仍需用户重新选择 File，这是浏览器安全限制下的预期行为         |
| 文件夹拖入通过 `webkitGetAsEntry` 递归时会丢相对路径              | 内部改为 `{ file, relativePath }` 收集结构，递归时按目录层级生成相对路径                              | 不支持目录枚举的浏览器仍只能降级为多文件选择                            |
| 预览阶段原本没有按每条编辑任务校验图片数量上限                    | 预览兼容性检查逐条调用现有图片引用校验，提前禁用确认并展示现有模型限制提示                            | 未做 Tauri 原生命令级文件夹选择；桌面 WebView 走浏览器文件/目录能力路径 |
| Playwright 上传命令只能访问工作区内文件                           | 将临时验证图片放到 `.playwright-cli/`，该目录已被 `.gitignore` 忽略                                   | 验证素材和截图不进入 git diff                                           |

## 后续建议

- P1 优先做 JSON/CSV 与已上传图片按文件名或相对路径匹配，补齐外部批量任务导入。
- 如果桌面端需要比 WebView 更稳定的文件夹权限和跨会话本地引用恢复，应新增 `src/lib/desktop-runtime.ts` 封装和对应 Rust command。
- 批量结果分组、前后对比和批量导出应与历史/资产库一起设计，避免在输出区临时堆状态。
