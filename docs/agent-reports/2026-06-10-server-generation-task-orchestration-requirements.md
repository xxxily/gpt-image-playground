## 后续建议

- 后续 agent 应继续 Phase 1 的未完成验收：启动可用 Docker daemon 后运行 Hatchet local control plane，获取 local profile/token，注册真实 Hatchet worker，执行 mock task，并记录 worker crash 与 dynamic concurrency/rate limit 映射证据；通过后再把 Phase 1 改为 `Accepted`。
- 不要在当前 App 内实现 Hatchet worker、持久队列或 provider adapter；当前 App 只实现接入配置、策略解析、状态同步和结果导入。
- P0 开发顺序建议严格遵守第 17 节：Phase 1 live Hatchet 验证补齐后，再进入 Phase 2 任务服务领域 API 与本地文件资产存储；当前 App 后台和用户侧接入分别属于 Phase 3、Phase 4。
- 每个阶段完成后必须回写需求文档第 17 节状态，并更新 `docs/agent-reports/` 下对应报告，避免后续 agent 误判进度。
