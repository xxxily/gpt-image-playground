## 问题与解决

| 问题                                                         | 解决办法                                                                                                                                                                                                                                               | 剩余风险                                                                          |
| ------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------- |
| 旧报告仍保留“用户 Key 委托待确认、先 PoC 再选底座”等过期判断 | 已将报告和需求文档改为最新确认结论：允许用户 Key 临时委托，底座选择 Hatchet，PoC 变成实施前置验收而不是选型开放项。                                                                                                                                    | 后续 agent 必须以需求文档第 21 节为准，避免沿用旧报告或旧讨论假设。               |
| Phase 1 live Hatchet 验证一度受阻                            | 初次 `hatchet server start` 因默认 Docker daemon 不可达失败；后续使用 Lima Docker socket 运行本地 Hatchet profile `gpt-image-task-phase1`，用子进程读取 profile 并注入 `HATCHET_CLIENT_TOKEN`、API URL、gRPC host/port 和 TLS strategy，未打印 token。 | 阻塞已解除；后续仍需避免在日志、报告或命令输出中打印 Hatchet token。              |
| 本地文件资产存储是 P0 首选但不是生产化多实例方案             | Phase 2 用任务服务本地文件系统保存 mock 结果，并在 manifest 中返回受控下载 URL、sha256、MIME、size、expiresAt；capabilities 预留 S3 兼容配置字段但标记 `s3CompatibleAvailable: false`。                                                                | 单节点本地文件系统仍有容量、备份和跨实例访问限制，已列入 Phase 7 增强。           |
| 自动重试可能消耗用户 Key 余额                                | Phase 2 默认关闭自动重试，并通过 `GET/POST /v1/admin/retry-policy` 返回供应商费用风险警告；手动重试会产生新 attempt 并保留任务事件。                                                                                                                   | Phase 5 仍需在当前 App 管理 UI、审计和权限层显式展示费用风险与操作记录。          |
| 管理摘要可能泄漏敏感内容                                     | Phase 2 admin task summary 测试覆盖不返回 `keyEnvelope`、原始 prompt 或 provider URL；Hatchet payload 测试仍覆盖不发送 credential、原始 prompt、provider URL 或 inputAssets。                                                                          | Phase 5 仍需补齐当前 App 管理员/超管可见性分层、审计和 URL 安全测试。             |
| 当前 App 与任务服务边界容易被后续实现打穿                    | Phase 2 实现范围限制在独立 `gpt-image-task-service/`；需求文档继续写明当前 App 只能通过 HTTP contract 接入，不能 import 任务服务内部 runtime、Hatchet adapter、asset store 或 storage 配置模型。                                                       | Phase 3-4 实现时仍需检查 import 边界和 Web/Tauri 分支，避免为了复用类型而跨边界。 |

- 下一步进入 Phase 3：在当前 App 中实现任务服务配置模型、端点接管策略模型、resolver、健康检查与 fallback；不要 import `gpt-image-task-service/` 内部 runtime、Hatchet adapter 或 asset store。
