## 问题与解决

| 问题                                                         | 解决办法                                                                                                                                                       | 剩余风险                                                                           |
| ------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| 旧报告仍保留“用户 Key 委托待确认、先 PoC 再选底座”等过期判断 | 本轮将报告和需求文档改为最新确认结论：允许用户 Key 临时委托，底座选择 Hatchet，PoC 变成实施前置验收而不是选型开放项。                                          | 后续 agent 必须以需求文档第 21 节为准，避免沿用旧报告或旧讨论假设。                |
| P0 API 示例原先包含视频、图生文等前向枚举，容易误导实现范围  | 将 `ManagedGenerationTaskRequest.taskType` 的 P0 示例收窄为 `image.generate` 和 `image.edit`，并说明其他任务类型必须通过后续 capabilities 和文档更新进入协议。 | 后续真正扩展视频/批量时要同步更新 API contract、capabilities 和验收标准。          |
| 结果资产存储容易被误解为当前 App 管理或对象存储优先          | 文档明确 P0 首选任务服务本地文件系统，当前 App 不管理任务服务内部存储；S3 是后续增强且由任务服务自身可视化配置。                                               | 单节点本地文件系统会带来容量、备份和跨实例访问限制，已列入风险和后续 S3 阶段。     |
| 自动重试可能导致用户 Key 余额被消耗                          | 文档要求失败默认用户手动重试；自动重试必须由后台显式配置，并展示供应商费用和额度消耗警告。                                                                     | 真实实现仍需把重试次数、配额、审计和费用提示做成强约束，而不是只做 UI 文案。       |
| 当前 App 容易越界管理任务服务内部配置                        | 文档将当前 App 后台限定为接入配置、策略绑定、能力摘要和诊断入口；Hatchet、worker、存储、S3、provider adapter 等由任务服务自身管理。                            | 任务服务项目创建后需要继续坚持独立管理面，不能为了开发便利把内部配置塞回当前 App。 |

## 后续建议

- 后续 agent 应先执行 Phase 0：创建独立任务服务 ADR，冻结 Hatchet 版本、部署依赖、状态映射、临时凭证方案和本地文件资产方案。
- 不要在当前 App 内实现 Hatchet worker、持久队列或 provider adapter；当前 App 只实现接入配置、策略解析、状态同步和结果导入。
- P0 开发顺序建议严格遵守第 17 节：先独立任务服务 mock 闭环，再接当前 App 后台配置，最后做用户侧提交恢复和结果导入。
- 每个阶段完成后必须回写需求文档第 17 节状态，并更新 `docs/agent-reports/` 下对应报告，避免后续 agent 误判进度。
